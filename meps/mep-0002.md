## Resolution

Rejected. This design is the venv style of virtual environment — a
project-local root that you activate and deactivate — and mip is expected to
move toward conda-style environments instead. It also mixes uv-style idioms
(spec + lockfile, `add`/`remove`/`lock`/`sync`) with venv-style ones
(`activate`/`deactivate`, a project-local `.mip/`) in an awkward way, and
placing everything under an `env` subcommand makes these operations
second-class.

## Summary

Add project-level, lockfile-backed environments to mip — the MATLAB analog of
a `uv`/`pip` virtual environment. A project declares the mip packages it
needs in a `mipenv.yaml` spec, pins the resolved transitive closure in a
`mipenv.lock` lockfile, and installs into a project-local `.mip/` root
instead of the global one. A new `mip env` command group (`init`, `add`,
`remove`, `lock`, `sync`, `status`, `activate`, `deactivate`) manages the
environment. The feature is a thin, purely additive layer over the existing
installer: every `mip env` command runs the normal install/resolve/load code
with `MIP_ROOT` pointed at the project-local root.

## Motivation

mip installs everything into one global root (`~/.mip`), so two projects on
the same machine share one package set and there is no way to record — let
alone reproduce — the exact versions a project was run against.

The immediate motivation is [calkit](https://github.com/calkit/calkit)
integration ([calkit#1016](https://github.com/calkit/calkit/issues/1016)):
calkit wants MATLAB users to define reproducible environments **without
building Docker images**. Its only current MATLAB support is a Docker
environment kind that needs a license server and a built image; a native
`mip` environment kind would let it treat MATLAB packages the way it already
treats `uv`/`conda`/`pixi` — check the lockfile, sync the environment, run a
command inside it.

## Proposal

### Files

Everything lives in the project directory (the folder `--directory` points
at, or the current folder):

| File          | Role                                                           | Format | Edited by |
|---------------|----------------------------------------------------------------|--------|-----------|
| `mipenv.yaml` | Spec: dependencies + channels                                  | YAML   | you       |
| `mipenv.lock` | Lock: resolved versions, `.mhl` URLs, SHA-256                  | JSON   | mip       |
| `.mip/`       | Project-local install root (`packages/`, cache, install state) | —      | mip       |

`mipenv.yaml` and `mipenv.lock` are committed; `.mip/` is gitignored.

```yaml
# mipenv.yaml
name: my-analysis

dependencies:
  - findtria                       # bare name -> resolved against core + channels
  - chebfun@1.0.0                  # pin a version
  - mip-org/labs/treeweave         # fully qualified: pin the channel

channels:                          # extra channels (besides mip-org/core)
  - mip-org/labs                   # consulted, in order, for bare names
```

The lock records the **full transitive closure** in dependency-first order.
Each entry carries the package's FQN, resolved version and architecture, the
exact `.mhl` URL and its SHA-256, and `source_hash`/`commit_hash` for
traceability; `direct: true` marks packages named in the spec. `sync`
therefore reinstalls from the lock with **no** network resolution.

### Commands

All under a new `mip env` group. Every subcommand accepts
`--directory <dir>`, so one project can hold several environments in
different subdirectories.

```
mip env init [--directory <dir>] [--name <name>]     Create mipenv.yaml + .mip
mip env add <pkg> [...] [--channel <c>] [--no-sync]  Add deps, re-lock, install
mip env remove <pkg> [...] [--no-sync]               Remove deps, re-lock, prune
mip env lock [--directory <dir>]                     Resolve spec -> mipenv.lock
mip env sync [--directory <dir>] [--relock]          Install exactly the lock
mip env status [--directory <dir>]                   Show declared/locked/installed
mip env activate [--directory <dir>] [--no-load]     Use env in this session
mip env deactivate                                   Return to the global root
```

Correspondence to uv: `add` ≈ `uv add`, `remove` ≈ `uv remove`,
`lock` ≈ `uv lock`, `sync` ≈ `uv sync`. `activate` has no uv analog because
mip runs *inside* a MATLAB session: it sets the project root for the rest of
the session and loads the direct packages onto the path.

### Design

The key observation is that `mip.paths.root()` already honors the `MIP_ROOT`
environment variable, so a project environment is just a project-local root
at `<project>/.mip`:

- `mip.env.with_root(root)` temporarily sets `MIP_ROOT` (restored on scope
  exit). Under that guard, `mip.install`, the resolver, the channel-index
  cache, and the per-project `directly_installed.txt` all operate on the
  project environment, isolated from the global one.
- `mip.env.resolve_lock` reuses the same machinery `install` uses
  (`mip.channel.fetch_index`, `mip.resolve.build_package_info_map`,
  `mip.dependency.build_graph`, `topological_sort`) to compute the pinned
  closure without installing.
- `mip.env.sync` installs each locked entry directly from its `mhl_url`,
  verifying `mhl_sha256`.

The only change to existing code is one dispatch case in `mip.m`; everything
else lives in a new `+mip/+env/` package.

### calkit integration

calkit models every environment as `{kind, path, ...}` in `calkit.yaml`, with
the lockfile a sibling of `path` (uv → `uv.lock`, pixi → `pixi.lock`). A
native mip kind fits directly (`kind: mip`, `path: mipenv.yaml` →
`mipenv.lock`). Because mip runs inside MATLAB, calkit invokes it via
`matlab -batch`, which it already does for its Docker MATLAB kind:
`mip env sync` to build/check the environment, and
`mip env activate` followed by the user's command to run inside it.

## Backward compatibility

Purely additive. Nothing in the global install path changes; without
`mip env`, behavior is identical to today.

## Alternatives

- **Spec named `mip.toml` (uv-style `pyproject.toml` analog).** mip already
  uses `mip.yaml` (package manifest) and `mip.json` (installed-package
  metadata); a third meaning for the `mip.*` prefix would be confusing. The
  distinct basename `mipenv.*` reads as "mip environment" and parallels
  `venv`/`virtualenv`. The lock is JSON (despite the `.lock` extension,
  following `uv.lock`) so it round-trips through MATLAB's native
  `jsonencode`/`jsondecode` with no extra dependency.
- **Docker-based reproducibility (calkit's current MATLAB kind).** Requires a
  license server and a built image; heavyweight for what is ultimately a set
  of MATLAB path entries.

## Open questions

- **Cross-platform locks.** The lock records the current architecture's
  `.mhl` URL + SHA. `sync` on a different arch re-resolves that package from
  its channel at the locked version to get the right binary. A fuller
  solution would lock all architectures at once (as `uv.lock` does for
  wheels); the lock format has room to grow into that.
- **Floating versions.** Branch "versions" like `main`/`master` are not
  immutable; the lock records `source_hash`/`commit_hash` for traceability
  but cannot yet pin an install to a specific commit (needs channel support).
- **Project discovery** does not walk up the directory tree; the project is
  either explicit (`--directory`) or the current folder.
- **Session load state** (`MIP_LOADED_PACKAGES`) is global to the MATLAB
  session, so activating a second environment in the same session mixes
  paths. Restart MATLAB (or `mip env deactivate`) between environments.
