## Summary

mip lives *inside* the MATLAB REPL; uv lives *outside* the Python
interpreter. That placement is the deepest structural difference between the
two tools, and it caps what mip can offer: from inside a session, mip can
never choose which MATLAB to run, never provide a truly clean-slate
execution, and never do anything at all before MATLAB (and a license) is up.

This MEP gives mip a second face: a `mip` executable on the user's PATH,
sharing the same root, formats, and (crucially) implementation with the
in-session tool. From the terminal,

```sh
mip run myscript.m
```

prepares the script's environment (project `mipenv.yaml`/`mipenv.lock` per
MEP 2, or inline metadata per MEP 4), selects and launches an appropriate
runtime, and runs the script inside it — uv ergonomics, one shot. And
because the CLI stands outside any particular interpreter, "an appropriate
runtime" need not be MATLAB: the same command can target
[GNU Octave](https://octave.org) or [numbl](https://numbl.org), running mip
packages with no MATLAB license at all.

The CLI is delivered in two phases that never fork the codebase: first a
thin shim that routes every command through `matlab -batch "mip …"` (one
implementation, slow but correct), then native fast paths for the provably
dumb operations only — lock-driven sync, runtime discovery, process launch.
Resolution logic is never reimplemented.

## Motivation

Each of the following is impossible from inside a running session, and each
is routine for uv:

- **Runtime selection.** MEP 4's `matlab: ">=R2021a"` field is "a check, not
  a provision" precisely because in-session mip is already inside the MATLAB
  it would need to choose. A shell-side `mip run` can discover the installed
  MATLABs (many users have several), pick one that satisfies the script's
  requirement, and launch it — and, beyond version selection, substitute a
  different MATLAB-compatible runtime entirely (Octave, numbl). mip still
  cannot *install* MATLAB (licensing), but choosing among installed runtimes
  covers most real cases, and the license wall itself disappears for the
  open runtimes.
- **Hermetic runs.** MEP 3's open questions — leaked `javaaddpath` entries,
  locked MEX files, `parpool` workers — all stem from restoring state inside
  a live session. A fresh process per run makes those problems vanish rather
  than get managed.
- **Environment preparation without MATLAB.** Syncing from `mipenv.lock` is
  dumb work: download each `mhl_url`, verify `mhl_sha256`, extract. None of
  it needs MATLAB. A CI job could build the entire environment without
  consuming a MATLAB license, checking one out only for the actual run.
- **One-shot invocation for everything else.** calkit hooks, CI jobs,
  READMEs, and shell-native newcomers currently quote MATLAB expressions
  into `matlab -batch "…"` strings. `mip run myscript.m` replaces that.

The nearest precedent is Julia: Pkg lives inside the REPL (like mip), and
Julia's answer to `uv run` is `julia --project=. script.jl` — flags on the
interpreter binary. mip cannot take that route because MATLAB's binary is
not mip's to extend; a wrapper CLI is the only way to get an outside face.

## Proposal

### Two faces, one truth

The CLI and the in-session tool share everything: the root (`~/.mip`), the
package tree, install state, channel formats, and the environment formats of
MEP 2. Neither owns a private cache or state file. The in-session tool
remains the primary interface for interactive work; the CLI is the interface
for scripts, CI, and tools.

Some verbs are inherently one-sided:

| Command                                   | In-session | CLI |
|-------------------------------------------|:----------:|:---:|
| `install`, `uninstall`, `update`, `list`, `info`, `avail` | yes | yes |
| `env init/add/remove/lock/sync/status`    | yes        | yes |
| `run`                                     | yes (scoped, MEP 3) | yes (fresh process) |
| `compile`, `test`, `bundle`               | yes        | yes (launches a runtime) |
| `load`, `unload`, `reset`, `env activate/deactivate` | yes | — (no live session; clear error) |
| runtime discovery/selection               | —          | yes |

### The smart/dumb split

The resolver — name resolution, version selection, dependency graphs — exists
once, in MATLAB, and stays there. The lockfile is the interface between the
smart half and the dumb half:

- **Smart (MATLAB-side only):** `env lock`, dependency resolution, anything
  consulting channel indexes to make choices.
- **Dumb (eligible for native CLI implementation):** consuming a lock
  (download, verify, extract), discovering runtimes, constructing a path,
  launching a process.

The CLI never gains a second resolver. If an operation requires resolution,
the CLI either routes it through a MATLAB-compatible runtime or refuses with
instructions.

### Phase 1: the shim

The bootstrap installer writes `~/.mip/bin/mip` (POSIX shell script; a
`mip.cmd` sibling on Windows) and suggests adding it to PATH. The shim
locates MATLAB and routes everything:

```
mip <args...>   →   matlab -batch "mip <args...>"
```

This is slow — every invocation pays MATLAB startup — but it is *correct by
construction*: there is exactly one mip implementation, so the shim can never
drift from it. It already delivers the one-shot UX (`mip run`, `mip env
sync`) for CI and calkit, and it requires only `matlab -batch` (R2019a+).

### Phase 2: native fast paths

Selected dumb operations get native implementations in the CLI, in priority
order:

1. **`mip env sync`** — consume `mipenv.lock` directly: download, SHA-256
   verify, extract into the project root. No runtime launched, no license
   consumed. This is the CI workhorse.
2. **Runtime discovery** — find installed MATLABs (standard install
   locations, `PATH`, the Windows registry), Octave, and numbl; report via
   `mip runtimes` (name pending); select per the rules below.
3. **`mip run` without in-session mip** — for a synced environment, the CLI
   can construct the path itself (the lock plus each installed package's
   `mip.json` lists every path entry) and launch the runtime with that path
   (`MATLABPATH`, Octave's `--path`, numbl's equivalent), so the script
   needs nothing mip-related inside the session at all.

### Runtime selection

```
mip run myscript.m                       # default runtime (MATLAB)
mip run myscript.m --runtime octave
mip run myscript.m --runtime numbl
```

- **MATLAB.** Discovery across installed versions; if the script or project
  declares `matlab: ">=R2021a"` (MEP 4), the CLI picks a satisfying
  installation rather than merely checking the current one — upgrading that
  field from a check toward uv-style interpreter selection.
- **numbl.** Already half-inside mip's model: mip's own source special-cases
  numbl (`isnumbl` checks; JVM-free operation), the architecture family
  `numbl_linux_x86_64` / `numbl_macos_arm64` / `numbl_wasm` exists with
  `numbl_wasm` as the universal fallback, and channel packages (fmm2d,
  fmm3d, finufft, jigsaw, …) already ship WASM-compiled MEX for it. The CLI
  launches it via numbl's own CLI (`numbl run` / `npx numbl`). A numbl run
  is fully license-free, end to end.
- **Octave.** Prospective. Much pure-MATLAB package code will run, but
  Octave's MEX ABI is incompatible with MATLAB's (it would need its own
  architecture family, which channels do not build today), and language
  compatibility varies by package. Support likely starts restricted to
  `any`-architecture packages, with the rest an open question.

One consequence the runtime dimension makes concrete: a lock resolved for
MATLAB on `macos_arm64` pins different artifacts than the same spec resolved
for numbl (`numbl_macos_arm64` / `numbl_wasm`). This is MEP 2's
"cross-platform locks" open question with a second axis — the lock format
needs room for per-runtime, per-arch artifact sets.

## Backward compatibility

Purely additive. The in-session tool changes only by gaining awareness that
some commands may arrive via `-batch` routing (already true today). Nothing
about channels, formats, or existing commands changes. Users who never add
`~/.mip/bin` to PATH are unaffected.

## Alternatives

- **A full native CLI (uv-style rewrite in Rust/Go).** Maximal speed,
  minimal startup — and a second implementation of the resolver that *will*
  drift from the MATLAB one. mip's identity is a pure-MATLAB tool; the
  phased shim keeps one source of truth and buys speed only where speed
  lives (sync, launch).
- **A Python wrapper.** Trades MATLAB startup for a Python dependency — the
  exact dependency situation uv exists to escape.
- **Interpreter flags instead of a CLI** (the Julia route). Not available
  for MATLAB. Partially available for numbl — numbl could natively learn
  `numbl run --project` — but that covers one runtime and still leaves
  environment sync needing a home.
- **A warm-session daemon** to amortize MATLAB startup (engine API holding a
  persistent session). Rejected for this MEP: it reintroduces exactly the
  shared-state problems the fresh-process model solves. Could be revisited
  separately if shim latency proves painful interactively.

## Open questions

- **Phase-2 implementation language.** POSIX sh + PowerShell keeps
  dependencies at zero but limits how much "dumb" logic is practical
  (SHA-256, JSON parsing are doable but grim). A small static binary is
  pleasant but is a second (if resolver-free) codebase. Node is already
  implied for the numbl runtime, but requiring it for the mip CLI couples
  the CLI to one runtime's toolchain.
- **Per-runtime locks.** Extend `mipenv.lock` with per-runtime/per-arch
  artifact sets (the `uv.lock`-locks-all-wheels model), or re-resolve on
  runtime mismatch as MEP 2 does for arch mismatch today?
- **Runtime requirement metadata.** Should MEP 4's `matlab:` field
  generalize (e.g. `runtimes: {matlab: ">=R2021a", numbl: ">=0.3"}`), and
  should packages be able to declare runtime compatibility in `mip.yaml`?
- **Default runtime.** Always MATLAB, or fall back to numbl/Octave when no
  MATLAB is installed (zero-license quickstart)?
- **Octave architecture family.** Is `any`-only Octave support useful enough
  to ship, and is a channel-built `octave_*` MEX family worth its CI cost?
- **Shim versioning.** The shim should stay dumb enough to almost never need
  updating, but phase-2 binaries version against lock formats — pin them to
  the in-tree mip version, or version independently?
