## Summary

Some packages are not libraries you build on but *tools* you point at things:
formatters, exporters, doc generators, test runners. This MEP gives mip the
analog of `uvx`/`pipx`/`npx` for them: `mipx`, which — like every mip command
after MEP 6 — exists on both faces, as a function in the MATLAB session and
as a command at the shell. In a session, with a figure open,

```matlab
mipx export_fig out.png -m2
```

runs export_fig's entry point in a cached environment containing *only*
export_fig and its dependencies — nothing is installed into your session's
package set, your project environment is neither consulted nor touched, and
when the call returns your path is exactly what it was. From the terminal,
the same runner serves tools that operate on files rather than on session
state (export_fig needs a live figure; a formatter or test runner does not):

```sh
mipx mbeautifier src/solver.m
mipx moxunit tests/ --runtime octave      # zero-license CI test run
```

Two pieces make this work: an **entry-point declaration** in `mip.yaml`
(`tools:` — the analog of Python's `console_scripts` and npm's `bin`), and a
**runner** (`mipx`, also spelled `mip tool run`) that resolves a tool name to
its package, materializes the tool's own environment, and calls the entry
point with shell-style string arguments. An optional `mip tool install`
makes a tool persistent.

## Motivation

The MATLAB ecosystem has real tools with real users: matlab2tikz (1.7k★),
export_fig (1.3k★), MBeautifier, m2html, MOxUnit/MOcov, matlab-schemer,
Dynare (the standard DSGE modeling tool in economics, driven entirely as
`dynare model.mod`). Today each of them is installed *into* the user's
environment: its functions and its dependencies sit on the path alongside
project code, permanently. Project environments (MEP 2) sharpen the problem
rather than solve it — a formatter is not a dependency of your research
project, but the only ways to use one inside an environment are to declare
it as one or to inject it per-run with `--with`, which MEP 3 already notes
leaves unrecorded packages in the project root.

`mip run` is close but not the right shape. The honest equivalence is

```
mipx <tool> <args...>   ≈   mip run --with <tool> "<entrypoint>('<arg1>', ...)"
```

*plus policy*: `mipx` ignores the project environment instead of running
inside it (the tool's version is not pinned by your project's lock, and the
tool's dependency closure never touches your project); it takes shell-style
*arguments* rather than a quoted MATLAB *expression*; and it resolves the
package's *declared* entry point rather than whatever function the caller
happens to name.

There is also a cautionary tale in the gap this fills: arguably the best
MATLAB linter/formatter, miss_hit, is written in Python — distributed via
pip/pipx because MATLAB offers tool authors no way to ship a runnable tool.
Python's and JavaScript's tool cultures exist in part *because*
`pipx`/`npx` made packages-as-tools a first-class category. This MEP is
partly a bet that the convention creates the category; the motivation is
candid that MATLAB's current tool bench is thinner than Python's.

## Proposal

### Entry points in `mip.yaml`

A package declares its runnable surface in a `tools:` map from tool name to
function:

```yaml
name: export_fig
tools:
  export_fig: export_fig          # tool name -> entry-point function
```

The function may be namespaced, and the tool name need not match the package
name:

```yaml
name: mbeautifier
tools:
  mbeautifier: MBeautify.formatFile
```

A package may declare several tools. As a fallback, a package with **no**
`tools:` map is still runnable if it exports a function whose name equals
the package name — which is exactly export_fig's shape, so `mipx export_fig`
works with no manifest change at all. (Whether to keep this fallback is an
open question; explicit is safer, implicit covers the existing ecosystem.)

`tools:` is carried through to `mip.json` at install/build time, so tool
resolution never needs the original manifest.

### Invocation and argument semantics

```
mipx <tool> [args...]
mipx <tool>@<version> [args...]
mipx <owner>/<channel>/<tool> [args...]
```

Each argument token is passed to the entry point as a char array — which is
precisely MATLAB's *command syntax*. `mipx export_fig out.png -m2` calls
`export_fig('out.png', '-m2')`, and any tool already usable in command
syntax (export_fig, Dynare, moxunit_runtests) is therefore mipx-compatible
with zero changes. Tools wanting numbers parse them (`str2double`), exactly
as command-syntax functions do today.

`mipx` is an alias for `mip tool run`, following uv's `uvx` = `uv tool run`.

### The tool environment

Running a tool materializes (or reuses) a cached environment containing the
tool's package and its resolved dependencies, keyed by FQN, version, and
architecture, under `<root>/tool-envs/` — the same mechanics as MEP 4's
script environments, with the same eviction question. Policy, matching uvx:

- **The project environment is ignored.** A `mipenv.yaml` in the current
  directory plays no role; your project's lock does not pin the tool's
  version, and running a tool records nothing in your project. Likewise,
  MEP 6's shell-session load state plays no role in (and is unaffected by)
  a tool run.
- **The session's package set is untouched.** In-session `mipx` loads the
  tool's environment *scoped* (MEP 3's `onCleanup` guard around path and
  load state), calls the entry point, and restores. Your loaded packages,
  your figures, and your workspace are as before — except for whatever the
  tool did to its target.

### Two faces, and which tools fit which

Following MEP 6, `mipx` exists in the session and on the shell, and the tool
category splits accordingly:

- **Session-context tools** operate on live session state. export_fig is
  the canonical example: it exports the *current figure*, so it only makes
  sense in-session — from a terminal there is no figure. matlab2tikz and
  matlab-schemer (MATLAB preferences) are the same shape. In-session `mipx`
  is what serves these: the tool appears, acts on your figure, disappears.
- **File-context tools** operate on files and directories: MBeautifier,
  m2html, MOxUnit, Dynare. These work identically in-session and from the
  terminal, where `mipx` (via MEP 6) launches a fresh runtime — and MEP 6's
  `--runtime` applies, so `mipx moxunit tests/ --runtime octave` is a test
  run with no MATLAB license (MOxUnit already supports Octave).

Shell `mipx` invoking a session-context tool cannot work in general; the CLI
should say so rather than export an empty figure (open question below).

### Persistent tools: `mip tool install`

The pipx-install analog, for tools used often enough that per-run
materialization is noise:

```
mip tool install mbeautifier      # persistent tool env + shell shim
mip tool list
mip tool uninstall mbeautifier
```

`install` pins the tool env as non-evictable and (on the shell side, per
MEP 6) writes a shim — `~/.mip/bin/mbeautifier` — so the tool is a plain
terminal command. Session-context tools get the persistent env but no shim.

## Backward compatibility

Purely additive: `tools:` is optional and ignored by older mip versions;
`mipx`/`mip tool` are new names; no existing command or format changes.
Packages never touched by this MEP behave exactly as today.

## Alternatives

- **Just use `mip run --with`.** The equivalence above shows what's lost:
  isolation from the project (both directions), argument ergonomics versus
  quoting MATLAB expressions in the shell, and declared entry points versus
  caller-supplied function names.
- **Implicit entry points only** (npm runs the `bin` matching the package
  name; the fallback rule alone). Covers export_fig and Dynare, but not
  MBeautifier (`MBeautify.formatFile`) or any package whose public face
  isn't its own name; and "every package whose name collides with one of
  its functions is silently runnable" is a weak contract.
- **A separate standalone tool** (as pipx is separate from pip). mip
  already owns both faces after MEP 6; a subcommand plus the `mipx` alias
  adds no install burden.

## Open questions

- **Keep the package-name fallback?** It makes the existing ecosystem
  (export_fig, dynare, m2html — all name-matching) runnable with zero
  manifest changes, at the cost of an implicit contract.
- **Session-context tools from the shell:** hard error, or attempt headless
  (some figure-context workflows do run under `-batch` with invisible
  figures)? Perhaps a `session: true` flag per tool in `tools:`.
- **Outputs and exit codes.** Shell `mipx` should map uncaught errors to a
  nonzero exit code (it does, via `-batch`). Should in-session `mipx`
  support capturing return values (`x = mipx(...)` functional form), or is
  display-only enough for tools?
- **Tool-env eviction** — shared with MEP 4's script-env cache question;
  one `mip env clean`-style answer should cover both.
- **Windows shims** for `mip tool install` (`.cmd` wrappers, PATH policy),
  building on MEP 6's `~/.mip/bin/` delivery.
