## Summary

Layer a declarative, lockfile-driven workflow — the MATLAB analog of `uv` —
on top of [MEP 8](mep-0008.md)'s environments. Where MEP 8 has you *manage*
an environment by hand, here you **declare** the dependencies you want and mip
**makes the environment match**, pinning the resolved closure in a lockfile so
any machine rebuilds it identically.

The spec is `mip.yaml` itself — the same file that is already the mip package
manifest — with package identity (`name:` / `version:`) made **optional**, so
a nameless `mip.yaml` is a plain project spec. It gains a `dev-dependencies`
group; `mip.json` remains the published projection (dev-deps and channels
stripped). The lock is **`mip.lock`**. New top-level project verbs —
`mip add`, `mip remove`, `mip lock`, `mip sync`, `mip run` — act on the
nearest `mip.yaml`, found by walking up from the current directory (as git
finds `.git` and cargo finds `Cargo.toml`).

The switch between MEP 8's hand-managed mode and this declarative one is a
single observable fact: **does `mip.lock` exist?** No mode flag, no setting.
Running `mip lock`, `mip add`, or `mip run` creates the lock and thereby opts
the directory in; committing the lock opts in the whole team.

## Motivation

[MEP 8](mep-0008.md) gives a project isolation but not reproducibility: delete
`.mip/` and the packages are gone, and nothing records exactly which versions
were installed. The workflow teams actually need is "clone the repo, run one
command, get the identical package set — on any machine, in CI, next year."

`uv` proved the ergonomics for Python: declare dependencies, lock them, sync
the environment, run inside it. This MEP brings that shape to MATLAB, with two
deliberate choices that distinguish it from the rejected [MEP 2](mep-0002.md):

- **One spec file, not two.** MEP 2 introduced a separate `mipenv.yaml`
  alongside the existing `mip.yaml` manifest, duplicating dependency
  declarations and giving the `mip.*` prefix a third meaning. Here the project
  spec *is* `mip.yaml`, with identity optional — no second file, no duplicated
  deps, and a repo can carry a `mip.yaml` forever without imposing this
  workflow on anyone.
- **Top-level verbs, not a subcommand ghetto.** MEP 2 buried
  `add`/`lock`/`sync` under `mip env`, making the everyday declarative
  operations second-class. Here they are top-level, peers of `mip install`.

The immediate external driver is unchanged from MEP 2: reproducible MATLAB
environments **without Docker** for
[calkit](https://github.com/calkit/calkit)
([calkit#1016](https://github.com/calkit/calkit/issues/1016)). A native mip
environment kind — lockfile-backed, no license server, no built image — lets
calkit treat MATLAB packages the way it already treats `uv`/`conda`/`pixi`.

## Proposal

### One spec file: `mip.yaml`

`mip.yaml` serves as **both** the packaging manifest and the project spec.
Package identity is optional, so the same file spans both uses:

```yaml
# mip.yaml — as a project spec (no identity)
dependencies:
  - findtria
  - chebfun@1.0.0
  - mip-org/labs/treeweave      # fully qualified: pin the channel

dev-dependencies:               # present for development, stripped from mip.json
  - moxunit

channels:                       # extra channels beyond mip-org/core
  - mip-org/labs
```

Add `name:` / `version:` and it is simultaneously a publishable package whose
own dependencies are the project's dependencies. The files involved:

| File       | Role                                                      | Format | Edited by |
|------------|-----------------------------------------------------------|--------|-----------|
| `mip.yaml` | Spec: dependencies, dev-dependencies, channels, `tools:`  | YAML   | you       |
| `mip.lock` | Lock: resolved versions, `.mhl` URLs, SHA-256             | JSON   | mip       |
| `.mip/`    | The project's local env (derived; see below)              | —      | mip       |

`mip.yaml` and `mip.lock` are committed; `.mip/` is gitignored. `mip.json`
stays what it is today — the installed/published projection — and CI builds it
from `mip.yaml`, **stripping `dev-dependencies` and `channels`**. Compared
with today, `mip.yaml` gains `dev-dependencies` and optional identity; existing
manifests are unaffected (see Backward compatibility). The lock is JSON despite
its `.lock` extension (following `uv.lock`), so it round-trips through MATLAB's
native `jsonencode`/`jsondecode` with no extra dependency.

### The two modes, and the switch

| | pip+venv mode ([MEP 8](mep-0008.md)) | uv mode (this MEP) |
|---|---|---|
| You… | manage the env by hand | declare dependencies |
| mip… | installs what you tell it | makes the env match the spec |
| Source of truth | the env's contents | `mip.lock` |
| `mip.lock` | absent | present |
| `.mip/` | the env itself | a disposable copy of the lock |
| Reproducible? | only via `mip freeze` | yes, by construction |

The mode is not a setting; it is the single observable fact of whether
`mip.lock` exists. `mip lock`, `mip add`, and `mip run` each create the lock
and thereby opt the directory into uv mode; committing the lock opts in the
repo for everyone who clones it.

In uv mode the lock is authoritative and `.mip/` is a disposable copy of it,
rebuilt identically by `mip sync` on any machine. `mip install` still works in
uv mode, but its additions are **temporary** — unrecorded in the lock and
removed by the next `mip sync`.

**Bridging from pip mode.** `mip init --from-env` writes a `mip.yaml` from the
active env (using MEP 8's `mip freeze`); or just run `mip add` once. The
first `mip sync` against a formerly hand-managed env lists what it would prune
and asks for confirmation before touching anything.

### Project discovery

Project commands act on the **project**: the nearest `mip.yaml`, found by
walking up from the current directory, the way git finds `.git` and cargo
finds `Cargo.toml`. The innermost spec wins; the walk stops at the home
directory and filesystem boundaries; `--directory <dir>` overrides it; and the
first output line always announces the target — `using project at
…/proj (mip.yaml)`. Activation state is ignored.

The one exception is `mip init`, which — like MEP 8's `mip env create` —
never walks: it creates a project *here*.

This is the deliberate counterpart to MEP 8's rule that **session** commands
never discover anything. The line that shows why both rules matter: the
session is pointed at a `scratch` env and the cwd is `./proj/tests`.

```matlab
mip install chebfun   % session command → installs into scratch (by session)
mip sync              % project command → walks up, finds ./proj/mip.yaml,
                      %   rebuilds ./proj/.mip, and announces "using project at ./proj"
```

`mip install` touches `scratch`; `mip sync` touches `./proj` and not
`scratch`. Only the committed, user-owned `mip.yaml` is ever discovered by
walking — never mutable env state.

### Commands

```
mip add <pkg> [...]      Edit spec → re-lock → sync         (--dev, --no-sync)
mip remove <pkg> [...]   Edit spec → re-lock → sync/prune   (--dev, --no-sync)
mip lock                 Resolve mip.yaml → write mip.lock   (--upgrade)
mip sync                 Make .mip/ match mip.lock exactly
mip run <target>         Re-lock if stale → sync → run scoped → restore  (--locked, --with)
```

- **`mip add` / `mip remove`** edit `mip.yaml`'s dependency list, then re-lock
  and sync. They enter uv mode if it isn't already in effect. `--dev` targets
  the `dev-dependencies` group; `--no-sync` edits and re-locks without
  installing.
- **`mip lock`** resolves `mip.yaml` and writes `mip.lock`; it installs
  nothing. The lock records the **full transitive closure** in
  dependency-first order — each entry carrying the package's FQN, resolved
  version and architecture, the exact `.mhl` URL and its SHA-256, and
  `source_hash` / `commit_hash` for traceability, with `direct: true` marking
  the packages named in the spec. `sync` can therefore reinstall from the lock
  with **no** network resolution. `--upgrade` re-resolves to the newest
  permitted versions. This reuses the machinery `install` uses today
  (`fetch_index`, `build_package_info_map`, `build_graph`, `topological_sort`).
- **`mip sync`** makes `.mip/` match `mip.lock` exactly, installing each locked
  entry directly from its `mhl_url` and verifying `mhl_sha256`. It errors if
  there is no lock. The first `sync` that would prune a formerly hand-managed
  env lists what it would remove and confirms first.
- **`mip run <target>`** re-locks if the spec is newer than the lock, syncs,
  activates the env **for the duration of the run only**, executes the target,
  and restores the session — the scoped run/restore model of
  [MEP 3](mep-0003.md), adopted here as a top-level uv-mode command reading
  `mip.yaml`/`mip.lock` and using the project walk. `--locked` forbids an
  implicit re-lock (for CI); `--with <pkg>` adds a package for this run only.
  See [MEP 3](mep-0003.md) for the detailed execution model and its open
  questions; this MEP changes only its spec/lock filenames and its discovery
  (project walk rather than `--directory`-only).

A typical uv-mode life cycle:

```matlab
mip init --env        % nameless mip.yaml (a project spec, no package identity)
mip add chebfun       % declare + lock + sync
mip activate          % work in it
% ...from a fresh clone:
mip sync              % rebuild .mip/ identically from mip.lock
% ...from the shell or CI:
% matlab -batch "mip run analysis.m"
```

### Refinement of `mip activate` and `mip status`

`mip activate` is defined in [MEP 8](mep-0008.md). This MEP refines its
**no-argument** case: rather than `./.mip` in the current directory only, it
uses the project walk — activating the nearest project's `.mip`. Explicit
`mip activate <name|path>` is unchanged.

`mip status` (also from MEP 8) gains its full reporting here: the mode
(pip+venv vs uv), the active env, the resolved project, the health of the
spec / lock / installed env, and any **drift** between them (spec newer than
lock, lock newer than `.mip/`). `--check` turns drift into a nonzero exit code
for CI.

### calkit integration

calkit models every environment as `{kind, path, ...}` in `calkit.yaml`, with
the lockfile a sibling of `path` (`uv` → `uv.lock`, `pixi` → `pixi.lock`). A
native mip kind fits directly: `kind: mip`, `path: mip.yaml` → `mip.lock`.
Because mip runs inside MATLAB, calkit invokes it via `matlab -batch`, as it
already does for its Docker MATLAB kind: `mip sync` to build or check the
environment, and `mip run <target>` to run inside it.

## Backward compatibility

Additive, with one small build-side change:

- **`mip.yaml`.** Existing manifests are unaffected: `dev-dependencies` and
  `channels` are new optional keys, and making `name:` / `version:` optional
  only *widens* what is accepted. A repo without a `mip.lock` is never in uv
  mode, so carrying a `mip.yaml` never opts anyone in.
- **`mip.json`.** CI's projection step must now strip `dev-dependencies` and
  `channels` when building `mip.json` from `mip.yaml`. Published packages are
  unchanged in shape.
- **Commands.** `add`, `remove`, `lock`, `sync` are new. `mip run` exists in
  the [MEP 3](mep-0003.md) draft; this MEP revises that draft's `mipenv.*`
  filenames to `mip.yaml`/`mip.lock` and its `--directory`-only targeting to
  the project walk. No existing command changes behavior.

## Alternatives

- **A separate spec file `mipenv.yaml` (the rejected [MEP 2](mep-0002.md)).**
  Duplicates dependency declarations, gives the `mip.*` prefix a third
  meaning, and can't sit in a repo without forcing the workflow. Folding the
  spec into `mip.yaml` with optional identity avoids all three.
- **A mode flag or config setting.** Rejected in favor of one git-visible
  fact: the lock's existence. Committing `mip.lock` is how a team opts in;
  deleting it is how they opt out. Nothing hidden in a config file.
- **Verbs under `mip env` (MEP 2's shape).** Made the everyday declarative
  operations second-class; promoted to top level here.
- **Name the lock `mipenv.lock`.** `mip.lock` pairs with `mip.yaml` and reads
  as "the lock for this project," matching `Cargo.toml`/`Cargo.lock` and
  `uv`'s `pyproject.toml`/`uv.lock`.

## Open questions

- **Does `mip activate` load by default?** Shared with [MEP 8](mep-0008.md);
  one flag settles it, `--load`/`--no-load` exist regardless.
- **Directory-walking in v1 or v2?** The walk is purely additive over a strict
  no-walk rule (which is a subset of it), so it can ship later behind
  `--no-walk` / a config opt-out if v1 should stay conservative.
- **First-sync prune confirmation UX.** Exact wording and behavior when a
  formerly hand-managed env first meets a lock and `sync` would prune.
- **Cross-platform and per-runtime locks.** The lock records the current
  architecture's `.mhl` URL + SHA; `sync` on a different arch re-resolves that
  package at the locked version. A fuller solution locks all architectures at
  once (as `uv.lock` does for wheels), and [MEP 6](mep-0006.md) adds a second
  axis (per-runtime artifacts for MATLAB / Octave / numbl). The lock format
  has room to grow into both.
- **Floating versions.** Branch "versions" like `main`/`master` are not
  immutable; the lock records `source_hash`/`commit_hash` for traceability but
  cannot yet pin an install to a specific commit (needs channel support).
- **Dev-dependencies semantics.** How `mip install .` (MEP 8) and `mip sync`
  treat the dev group, and whether dev-deps are meaningful only for the
  top-level project (never for a transitively-depended package).
- **Phase-2 CLI implementation language** and per-runtime lock-format details
  — carried over from [MEP 6](mep-0006.md).
