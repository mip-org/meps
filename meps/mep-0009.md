|----------------|--------------------------------------------------------------|

| **Status**     | Draft                                                       |
| **Created**    | 2026-07-10                                                  |
| **Requires**   | [MEP 8](mep-0008.md)                                        |

## Summary

Layer a declarative, lockfile-driven workflow ‚Äî the MATLAB analog of `uv` ‚Äî
on top of [MEP 8](mep-0008.md)'s environments. Where MEP 8 has you *manage*
an environment by hand, here you **declare** the dependencies you want and mip
**makes the environment match**, pinning the resolved closure in a lockfile so
any machine rebuilds it identically.

The spec is `mip.yaml` itself ‚Äî the same file that is already the mip package
manifest ‚Äî with package identity (`name:` / `version:`) made **optional**, so
a nameless `mip.yaml` is a plain project spec. It gains a `dependency_groups`
map (`dev` is the conventional group); `mip.json` remains the published
projection (groups and channels stripped). The lock is **`mip.lock`**. The
new verbs live in one command group, **`mip project`**:

```
mip project init / add / remove / lock / sync / run / status
```

They act on the nearest `mip.yaml`, found by walking up from the current
directory (as git finds `.git` and cargo finds `Cargo.toml`). Grouping them
keeps the everyday surface ‚Äî `mip install`, `mip load`, `mip env` ‚Äî exactly
as it is: a user who never adopts projects never meets these commands.

The switch between MEP 8's hand-managed mode and this declarative one is a
single observable fact: **does `mip.lock` exist?** No mode flag, no setting.
Running `mip project lock`, `add`, or `run` creates the lock and thereby opts
the directory in; committing the lock opts in the whole team.

## Motivation

[MEP 8](mep-0008.md) gives a project isolation but not reproducibility: delete
`.mip/` and the packages are gone, and nothing records exactly which versions
were installed. The workflow teams actually need is "clone the repo, run one
command, get the identical package set ‚Äî on any machine, in CI, next year."

`uv` proved the ergonomics for Python: declare dependencies, lock them, sync
the environment, run inside it. Early feedback on the mip prototype asked for
exactly this ‚Äî "`uv` for MATLAB, not `pip` for MATLAB"
([mip-org/mip#337](https://github.com/mip-org/mip/issues/337)). This MEP
brings that shape to MATLAB, staying as close to `uv`'s commands and
semantics as mip's model allows, with two deliberate choices that distinguish
it from the rejected [MEP 2](mep-0002.md):

- **One spec file, not two.** MEP 2 introduced a separate `mipenv.yaml`
  alongside the existing `mip.yaml` manifest, duplicating dependency
  declarations and giving the `mip.*` prefix a third meaning. Here the project
  spec *is* `mip.yaml`, with identity optional ‚Äî no second file, no duplicated
  deps, and a repo can carry a `mip.yaml` forever without imposing this
  workflow on anyone.
- **A dedicated `mip project` group, not an overloaded `mip env`.** MEP 2 put
  `add`/`lock`/`sync` under `mip env`, conflating two nouns ‚Äî the environment
  (a place packages live) and the project (a spec and its lock) ‚Äî and made
  that group the only way to manage environments at all. Here the declarative
  verbs live under `mip project`, named for the thing they act on, while
  MEP 8's `mip env` and the imperative verbs stay untouched. (`uv` itself
  uses top-level verbs, but `uv` is a standalone binary whose whole surface
  *is* the project workflow; mip's top level is already the imperative
  package manager, and a user who just wants `mip install` should never have
  to wonder what `mip add` does differently ‚Äî see Alternatives.)

The immediate external driver is unchanged from MEP 2: reproducible MATLAB
environments **without Docker** for
[calkit](https://github.com/calkit/calkit)
([calkit#1016](https://github.com/calkit/calkit/issues/1016)). A native mip
environment kind ‚Äî lockfile-backed, no license server, no built image ‚Äî lets
calkit treat MATLAB packages the way it already treats `uv`/`conda`/`pixi`.

## Proposal

### One spec file: `mip.yaml`

`mip.yaml` serves as **both** the packaging manifest and the project spec.
Package identity is optional, so the same file spans both uses:

```yaml
# mip.yaml ‚Äî as a project spec (no identity)
dependencies:
  - findtria
  - chebfun@1.0.0
  - mip-org/labs/treeweave      # fully qualified: pin the channel

dependency_groups:              # development-time; never published
  dev:
    - moxunit
  docs:
    - m2html

channels:                       # extra channels beyond mip-org/core
  - mip-org/labs
```

Add `name:` / `version:` and it is simultaneously a publishable package whose
own dependencies are the project's dependencies. The files involved:

| File       | Role                                                      | Format | Edited by |
|------------|-----------------------------------------------------------|--------|-----------|
| `mip.yaml` | Spec: dependencies, dependency_groups, channels (and `tools:`, see [MEP 7](mep-0007.md)) | YAML | you |
| `mip.lock` | Lock: resolved versions, `.mhl` URLs, SHA-256             | JSON   | mip       |
| `.mip/`    | The project's local env (derived; see below)              | ‚Äî      | mip       |

`mip.yaml` and `mip.lock` are committed; `.mip/` is gitignored. `mip.json`
stays what it is today ‚Äî the installed/published projection ‚Äî and CI builds it
from `mip.yaml`, **stripping `dependency_groups` and `channels`**. Compared
with today, `mip.yaml` gains `dependency_groups` and optional identity;
existing manifests are unaffected (see Backward compatibility). The lock is
JSON despite its `.lock` extension (following `uv.lock`), so it round-trips
through MATLAB's native `jsonencode`/`jsondecode` with no extra dependency.

### Dependency groups

`dependency_groups` is a map of named extra dependency lists ‚Äî `uv`'s shape
([PEP 735](https://peps.python.org/pep-0735/)), with `dev` as the
conventional group. `--dev` is shorthand for `--group dev` everywhere it
appears. Groups are development-time state for the top-level project:

- **`mip project lock`** resolves and locks **all** groups; each lock entry
  records which groups require it, so `sync` can select without re-resolving.
- **`mip project sync`** and **`run`** install `dependencies` **plus the
  `dev` group** by default ‚Äî clone, sync, run the tests just works, as with
  `uv sync`. `--no-dev` skips it, `--group <name>` adds another group,
  `--all-groups` installs every group.
- Groups are **stripped from `mip.json`**, so a published package never
  carries them ‚Äî they are meaningful only for the top-level project, by
  construction. Session commands are untouched: [MEP 8](mep-0008.md)'s
  `mip install .` reads `dependencies:` only.

### The two modes, and the switch

| | pip+venv mode ([MEP 8](mep-0008.md)) | uv mode (this MEP) |
|---|---|---|
| You‚Ä¶ | manage the env by hand | declare dependencies |
| mip‚Ä¶ | installs what you tell it | makes the env match the spec |
| Source of truth | the env's contents | `mip.lock` |
| `mip.lock` | absent | present |
| `.mip/` | the env itself | a disposable copy of the lock |
| Reproducible? | no ‚Äî the env's contents are the only record | yes, by construction |

The mode is not a setting; it is the single observable fact of whether
`mip.lock` exists. `mip project lock`, `add`, and `run` each create the lock
and thereby opt the directory into uv mode; committing the lock opts in the
repo for everyone who clones it.

In uv mode the lock is authoritative and `.mip/` is a disposable copy of it,
rebuilt identically by `mip project sync` on any machine. `mip install` still
works in uv mode, but its additions are **temporary** ‚Äî unrecorded in the lock
and removed by the next `mip project sync`. (This is also why MEP 8's
`mip env create` refuses its no-argument form when a `mip.lock` is present:
in uv mode `./.mip` is derived, and `mip project sync` owns it.)

**Bridging from pip mode.** `mip project init --from-env` writes a `mip.yaml`
by reading the active environment directly ‚Äî its directly-installed packages
become the dependency list, as names only with no version pins: the spec
declares *what*, and the first `mip project lock` pins exact versions.
Non-channel installs (local, File Exchange, web), which cannot be locked,
are skipped with a note. (MEP 8 keeps no other record of an env's
contents; its draft `mip freeze` command was cut in favor of this.)
Alternatively, just run `mip project add` once. The first `mip project sync`
against a formerly hand-managed env lists what it would prune and asks for
confirmation before touching anything.

### Project discovery

Project commands act on the **project**: the nearest `mip.yaml`, found by
walking up from the current directory, the way git finds `.git` and cargo
finds `Cargo.toml`. The innermost spec wins; the walk stops at the home
directory and filesystem boundaries; `--directory <dir>` overrides it; and the
first output line always announces the target ‚Äî `using project at
‚Ä¶/proj (mip.yaml)`. Activation state is ignored.

The one exception is `mip project init`, which ‚Äî like MEP 8's
`mip env create` ‚Äî never walks: it creates a project *here*.

This is the deliberate counterpart to MEP 8's rule that **session** commands
never discover anything. The line that shows why both rules matter: the
session is pointed at a `scratch` env and the cwd is `./proj/tests`.

```matlab
mip install chebfun   % session command ‚Üí installs into scratch (by session)
mip project sync      % project command ‚Üí walks up, finds ./proj/mip.yaml,
                      %   rebuilds ./proj/.mip, and announces "using project at ./proj"
```

`mip install` touches `scratch`; `mip project sync` touches `./proj` and not
`scratch`. Only the committed, user-owned `mip.yaml` is ever discovered by
walking ‚Äî never mutable env state.

### Commands

```
mip project init             Create a nameless mip.yaml here            (--from-env)
mip project add <pkg> [...]  Edit spec ‚Üí re-lock ‚Üí sync                 (--dev, --group, --no-sync)
mip project remove <pkg> ... Edit spec ‚Üí re-lock ‚Üí sync/prune           (--dev, --group, --no-sync)
mip project lock             Resolve mip.yaml ‚Üí write mip.lock          (--upgrade)
mip project sync             Make .mip/ match mip.lock                  (--no-dev, --group, --all-groups)
mip project run <target>     Lock if stale ‚Üí sync ‚Üí run scoped ‚Üí restore  (--locked, --no-sync, --with)
mip project status           Report project, mode, env health, drift    (--check)
```

`mip project` with no subcommand shows the resolved project ‚Äî an alias for
`mip project status`, mirroring how bare `mip env` shows the active env.

- **`mip project init`** creates a nameless `mip.yaml` (a project spec with no
  package identity) in the current directory. `--from-env` seeds the
  dependency list from the active environment, as described above. The
  existing `mip init` package scaffold is untouched; this is its project-spec
  sibling.
- **`mip project add` / `mip project remove`** edit `mip.yaml`'s dependency
  list, then re-lock and sync. They enter uv mode if it isn't already in
  effect. `--dev` / `--group <name>` target a dependency group; `--no-sync`
  edits and re-locks without installing.
- **`mip project lock`** resolves `mip.yaml` (all groups included) and writes
  `mip.lock`; it installs nothing. The lock records the **full transitive
  closure** in dependency-first order ‚Äî each entry carrying the package's
  FQN, resolved version and architecture, the exact `.mhl` URL and its
  SHA-256, `source_hash` / `commit_hash` for traceability, `direct: true` for
  the packages named in the spec, and the groups that require it. The digest
  and hashes are copied from the channel index entry ‚Äî locking downloads
  nothing ‚Äî so an index entry without a digest yields a lock entry without
  one, which `sync` installs unverified. The lock also records a hash of the
  `mip.yaml` it was resolved from, which is how staleness ("the spec is
  newer than the lock") is detected ‚Äî by content, robustly across fresh
  clones, rather than by file mtimes. `sync` can
  therefore reinstall from the lock with **no** network resolution.
  `--upgrade` re-resolves to the newest permitted versions. This reuses the
  machinery `install` uses today (`fetch_index`, `build_package_info_map`,
  `build_graph`, `topological_sort`).
- **`mip project sync`** makes `.mip/` match `mip.lock`, installing each
  locked entry directly from its `mhl_url` and verifying `mhl_sha256` when
  the lock carries one ‚Äî the
  `dependencies` closure plus the `dev` group by default (see Dependency
  groups for the flags). It creates `./.mip` (marker and all) if it does not
  exist, and errors if there is no lock. The first `sync` that would prune a
  formerly hand-managed env lists what it would remove and confirms first
  (`--yes` skips; the `MIP_CONFIRM` environment variable is honored, as in
  `mip env delete`). When the spec carries package identity (`name:`),
  `sync` finishes by installing the project itself into the env as an
  **editable** install ‚Äî the `uv sync` behavior ‚Äî and never prunes it; a
  nameless spec installs dependencies only.
- **`mip project run <target>`** ‚Äî lock if stale, sync, run the target inside
  the env, restore the session. Specified in full below.
- **`mip project status`** is the richer status reporting
  [MEP 8](mep-0008.md) explicitly deferred to this MEP: the mode (pip+venv vs
  uv), the active env, the resolved project, the health of the spec / lock /
  installed env, and any **drift** between them (spec newer than lock, lock
  newer than `.mip/`). `--check` turns drift into a nonzero exit code for CI.

A typical uv-mode life cycle:

```matlab
mip project init          % nameless mip.yaml (a project spec, no package identity)
mip project add chebfun   % declare + lock + sync
mip activate              % work in it
% ...from a fresh clone:
mip project sync          % rebuild .mip/ identically from mip.lock
% ...from the shell or CI:
% matlab -batch "mip project run analysis.m"
```

### Running code ‚Äî `mip project run`

`mip project run <target>` is the mip analog of `uv run`: it makes sure the
project is locked and synced, activates its env **for the duration of the run
only**, executes the target, and restores the session. Where `mip activate`
changes the session until further notice, `run` is scoped: after it returns,
the MATLAB path, load state, and mip root are exactly what they were before.
(This folds in and supersedes the earlier [MEP 3](mep-0003.md) draft, which
targeted the rejected MEP 2's machinery.)

A run proceeds in four steps:

1. **Lock.** If `mip.lock` is missing or stale (its recorded `mip.yaml`
   hash no longer matches the file), re-lock (as
   `mip project lock`) ‚Äî `uv run`'s behavior. With `--locked`, error instead:
   in CI, an implicit re-resolve should be a failure, not a convenience.
2. **Sync.** As `mip project sync` (dev group included by default), plus any
   `--with <pkg>` packages ‚Äî installed into the project env but recorded
   nowhere, so the next plain `sync` prunes them. `--with` is repeatable and
   accepts the same package syntax as `mip install`. `--no-sync` skips this
   step and runs against whatever is installed (erroring if a locked package
   is missing).
3. **Activate, scoped.** The same full swap as `mip activate --load` ‚Äî save
   the session state, point `MIP_ROOT` at the project env, load its direct
   packages plus the `--with` packages ‚Äî but guarded with `onCleanup` rather
   than left for the rest of the session.
4. **Run and restore.** Execute the target. On return ‚Äî normal or error ‚Äî
   restore the previous `MIP_ROOT`, MATLAB path, and load state, then rethrow
   any error.

`<target>` takes one of three forms, disambiguated syntactically:

- **A script** ‚Äî anything containing a path separator or ending in `.m`:
  executed with `run`, from the directory it lives in (MATLAB's usual script
  semantics).
- **A function call** ‚Äî a bare identifier, optionally followed by arguments:
  evaluated exactly as MATLAB command syntax, so `mip project run f 3` is
  `f('3')` and every argument arrives as char. For non-char arguments ‚Äî or
  arguments starting with `--`, which would parse as flags ‚Äî use the
  expression form.
- **An expression** ‚Äî anything else, in quotes: evaluated in a scoped,
  throwaway workspace.

Every form executes in a scoped workspace that is discarded when the run
returns; mip never injects variables into the base workspace (no
`evalin`/`assignin`). A target's output is what it displays or writes to
disk.

**Restoration is best-effort by design.** mip-managed state ‚Äî root, path,
load state ‚Äî is guaranteed via `onCleanup`, but a run can still leak
`javaaddpath` entries, globals, or loaded MEX files, and `parpool` workers do
not inherit the temporarily modified path. Full isolation is what
`matlab -batch` is for:

```sh
matlab -batch "mip project run analysis.m"
```

‚Äî which is also the shell/CI spelling. There the scoping is trivial (the
session ends anyway), and what `run` contributes is the lock/sync/load
preamble in one command.

### Refinement of `mip activate`

`mip activate` is defined in [MEP 8](mep-0008.md). This MEP refines its
**no-argument** case: rather than `./.mip` in the current directory only, it
uses the project walk ‚Äî activating the nearest project's `.mip`. Explicit
`mip activate <name|path>` is unchanged.

### calkit integration

calkit models every environment as `{kind, path, ...}` in `calkit.yaml`, with
the lockfile a sibling of `path` (`uv` ‚Üí `uv.lock`, `pixi` ‚Üí `pixi.lock`). A
native mip kind fits directly: `kind: mip`, `path: mip.yaml` ‚Üí `mip.lock`.
Because mip runs inside MATLAB, calkit invokes it via `matlab -batch`, as it
already does for its Docker MATLAB kind: `mip project sync` to build or check
the environment, and `mip project run <target>` to run inside it.

## Backward compatibility

Additive, with one small build-side change:

- **`mip.yaml`.** Existing manifests are unaffected: `dependency_groups` and
  `channels` are new optional keys, and making `name:` / `version:` optional
  only *widens* what is accepted. A repo without a `mip.lock` is never in uv
  mode, so carrying a `mip.yaml` never opts anyone in.
- **`mip.json`.** CI's projection step must now strip `dependency_groups` and
  `channels` when building `mip.json` from `mip.yaml`. Published packages are
  unchanged in shape.
- **Commands.** The `mip project` group is new and collides with nothing; the
  existing `mip init` package scaffold is untouched. The
  [MEP 3](mep-0003.md) draft (`mip run`) is folded in and **superseded**: its
  execution model lands here as `mip project run`, with the
  `mip.yaml`/`mip.lock` filenames and the project walk. No existing command
  changes behavior.

## Deferred

- **Commit pinning.** Branch "versions" like `main` are not immutable, and
  the lock cannot yet pin an install to a specific commit ‚Äî the lock records
  `source_hash`/`commit_hash` for traceability, but honoring them needs
  channel/install support for commit-addressed sources. A real adoption
  blocker (the MDOcean pilot needed unreleased OpenFLASH code that no
  published version satisfied), but orthogonal to the project layer: a
  separate MEP.
- **Toolbox requirements.** Declaring required MathWorks toolboxes in
  `mip.yaml` (mip cannot install them, but `status`/`sync` could check and
  report) ‚Äî also hit by the MDOcean pilot, also a separate MEP.
- **All-architecture locking.** v1 locks the current architecture's `.mhl`
  URL + SHA; `sync` on a different arch re-resolves that package at the
  locked version. Locking every architecture at once (as `uv.lock` does for
  wheels) is format growth the lock leaves room for.
- **Further `uv` verbs.** `mip project tree` (print the locked dependency
  tree) is the likely follow-on; `export` and `version` have no demonstrated
  need yet.
- **Inline script dependencies** (the PEP 723 / `uv run --script` analog ‚Äî
  dependency metadata embedded in a script, no project directory at all).
  `mip project run` is the execution vehicle it would plug into.
- **Per-runtime lock dimensions and the shell CLI** ‚Äî [MEP 6](mep-0006.md)'s
  territory (per-runtime artifacts for MATLAB / Octave / numbl, and the
  phase-2 CLI).

## Alternatives

- **Top-level verbs (`mip add`, `mip lock`, `mip sync`, `mip run`) ‚Äî `uv`'s
  own shape.** `uv` is a standalone binary whose entire surface is the
  project workflow, so top-level verbs are its natural spelling. mip's top
  level is already the imperative package manager (`install`, `uninstall`,
  `load`, `env`, ‚Ä¶); putting `add` beside `install` and `sync` beside
  `update` would present two overlapping vocabularies to every user ‚Äî
  including the majority who only want the basics ‚Äî and invite exactly the
  "what does `add` do that `install` doesn't?" confusion. One `mip project`
  group keeps the layer self-naming, discoverable in one place, and invisible
  until opted into.
- **A separate spec file `mipenv.yaml` (the rejected [MEP 2](mep-0002.md)).**
  Duplicates dependency declarations, gives the `mip.*` prefix a third
  meaning, and can't sit in a repo without forcing the workflow. Folding the
  spec into `mip.yaml` with optional identity avoids all three.
- **A mode flag or config setting.** Rejected in favor of one git-visible
  fact: the lock's existence. Committing `mip.lock` is how a team opts in;
  deleting it is how they opt out. Nothing hidden in a config file.
- **Group the verbs under `mip env` (MEP 2's shape).** The wrong noun: it
  conflates environments (where packages live) with projects (a spec and its
  lock), and MEP 2 made the group the only way to manage environments at all.
  `mip env` stays as MEP 8 defined it ‚Äî env objects only.
- **Name the lock `mipenv.lock`.** `mip.lock` pairs with `mip.yaml` and reads
  as "the lock for this project," matching `Cargo.toml`/`Cargo.lock` and
  `uv`'s `pyproject.toml`/`uv.lock`.
- **A scoped `mip activate` instead of `run`.** An activation that lasts only
  for the current scope isn't useful interactively, where
  activate/deactivate are typed at the prompt. The two serve different modes:
  `activate` for sessions, `run` for entry points.

## Resolved questions

Decisions reached in review, recorded for the discussion thread:

- **Dependency-group shape.** The group map (`dependency_groups`), `uv`'s
  PEP 735 shape, over a flat `dev_dependencies` list ‚Äî `dev` is the
  conventional group and `--dev` is shorthand for `--group dev`.
- **Dev-group defaults.** `sync` and `run` install the `dev` group by default
  (`--no-dev` / `--group <name>` / `--all-groups`), matching `uv sync`;
  `lock` locks all groups; all groups are stripped from `mip.json`; MEP 8's
  session-side `mip install .` reads `dependencies:` only.
- **Directory walking.** Ships in v1 ‚Äî `uv` and cargo both walk; the
  `using project at ‚Ä¶` announce line and the `--directory` override keep it
  unsurprising.
- **`mip run`.** Folded into this MEP as `mip project run`;
  [MEP 3](mep-0003.md) is superseded. The default follows `uv run` ‚Äî re-lock
  when the spec is newer than the lock ‚Äî with `--locked` for CI strictness.
- **Run targets.** Script path, quoted expression, or a bare function name
  with command-syntax arguments (`mip project run f 3` ‚Üí `f('3')`). All
  forms execute in a scoped, throwaway workspace ‚Äî mip never writes into
  the base workspace (no `evalin`/`assignin`).
- **`--with` packages** install into the project env, unrecorded, and the
  next plain `sync` prunes them ‚Äî no ephemeral overlay root.
- **Restore policy.** Best-effort: mip-managed state (root, path, load state)
  is guaranteed via `onCleanup`; Java class paths, globals, MEX locks, and
  `parpool` workers are documented limits, with `matlab -batch` as the full
  isolation story.
- **Lock architecture scope.** v1 locks the current architecture; other
  arches re-resolve at the locked version (see Deferred).
- **Commit pinning and toolbox requirements.** Both real (both surfaced by
  the MDOcean pilot), both separate MEPs (see Deferred).
- **First-sync prune.** List what would be removed and confirm; `--yes`
  skips, `MIP_CONFIRM` honored ‚Äî the `mip env delete` pattern.
- **Does `sync` install the project itself?** Yes, when the spec has
  identity: an editable install into `.mip/`, never pruned ‚Äî the `uv sync`
  behavior. A nameless spec installs dependencies only.
- **Where the locked SHA-256 comes from.** Copied from the channel index
  entry; `lock` downloads nothing. An index entry without a digest locks
  without one and installs unverified.
- **Staleness detection.** The lock records a hash of the `mip.yaml` it
  resolved; "spec newer than lock" is a content-hash mismatch, not an mtime
  comparison (which fresh clones would defeat).
- **`--from-env` entries.** Names only, no version pins ‚Äî the immediate
  `mip project lock` records exact versions. Non-channel installs are
  skipped with a note.
- **Extra verbs in v1.** None; `mip project tree` is the likely follow-on.
- **Bare `mip project`.** An alias for `mip project status`.
