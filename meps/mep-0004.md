## Summary

Let a single `.m` file declare its own environment — dependencies, channels,
and a minimum MATLAB version — in a specially marked comment block, the
MATLAB analog of Python's [PEP 723](https://peps.python.org/pep-0723/)
(implemented by `uv run --script`):

```matlab
% VORTEX_SHEDDING  Reproduce figure 3 of the paper.

% /// mip
% matlab: ">=R2021a"
% dependencies:
%   - chebfun@1.2.0
%   - findtria
% channels:
%   - mip-org/labs
% ///

u = chebfun(...);
```

`mip run vortex_shedding.m` ([MEP 3](mep-0003.md)) detects the block, builds
a cached ephemeral environment from it — no `mipenv.yaml`, no project
directory — and runs the script inside that environment, scoped as usual. A
script becomes a self-contained, shareable, reproducible unit: email it,
gist it, drop it in supplementary materials, and `mip run` is all a recipient
needs.

## Motivation

MEP 2 makes environments reproducible at the granularity of a *project
directory*: three artifacts (`mipenv.yaml`, `mipenv.lock`, `.mip/`) alongside
the code. That is the right shape for a project, and exactly the wrong shape
for the smallest unit of MATLAB sharing — the single script: a bug
reproduction attached to an issue, an example in a package's documentation, a
figure script in a paper's supplement, a homework starter file. Asking the
recipient to reconstruct a project around a file they were handed (or asking
the sender to ship a directory instead of a file) loses most of the audience.

Python had the same gap between projects and scripts; PEP 723 closed it with
inline metadata, and `uv run script.py` made the experience seamless. The
pieces mip needs are already in place after MEPs 2 and 3 — resolution,
ephemeral roots via `MIP_ROOT`, scoped execution. This MEP adds the last
piece: letting the script itself carry the spec.

## Proposal

### The metadata block

A metadata block is a contiguous run of comment lines delimited by
`% /// mip` and `% ///`:

- The block starts at a line whose content is exactly `% /// mip` and ends at
  the next line whose content is exactly `% ///`.
- Every line in between must begin with `%`; the metadata content is the
  remainder of each line after stripping the leading `% ` (or bare `%` for
  empty lines).
- The stripped content parses as YAML with the same syntax as `mipenv.yaml`
  (mip's existing YAML subset; no new parser).
- The block may appear in the *leading* comment region of the file (before
  the first executable line, including comments above a `function` line) or
  in a *trailing* comment block at the end of the file. At most one block per
  file; more than one is an error.

**Interaction with `help`.** MATLAB's `help` prints the file's first
contiguous comment block, so a metadata block at the very top would become
the script's help text. There are two ways to avoid the collision:

- *Help first.* Put the help text first, a blank line, then the metadata
  block; `help` stops at the blank line.
- *Block at the end of the file.* A trailing block can never interact with
  `help` (or with anything else MATLAB reads from leading comments), at the
  cost of not being the first thing a reader sees. Trailing metadata has
  precedent — Emacs file-local variables live in a comment block at the end
  of the file.

Both placements are valid; `mip env init --script` (below) uses the trailing
form by default.

### Fields

The block accepts the `mipenv.yaml` keys, plus one script-specific field:

| Field          | Meaning                                                        |
|----------------|----------------------------------------------------------------|
| `dependencies` | Packages, same syntax as `mipenv.yaml` (bare, `@version`, FQN) |
| `channels`     | Extra channels consulted for bare names, in order              |
| `matlab`       | Minimum MATLAB release, e.g. `">=R2021a"`                      |

`matlab` is a *check*, not a provision: mip cannot install MATLAB the way uv
installs Python interpreters. `mip run` compares the requirement against the
running session (`matlabRelease`) and errors up front with a clear message if
it isn't satisfied. The initial grammar is deliberately minimal — `>=` on a
release name — with room to grow.

### Execution

`mip run script.m` gains one step at the front of the MEP 3 pipeline: read
the script's leading comments. If a metadata block is present:

1. **Ephemeral environment.** Instead of `<project>/.mip`, the environment
   root is a cache directory under the *global* mip root, keyed by a hash of
   the normalized block content and the architecture:
   `<root>/script-envs/<hash>/`. Unchanged metadata reuses the cached
   environment across runs (fast path: no resolution, no network); edited
   metadata hashes to a fresh one.
2. **No project.** Any `mipenv.yaml` in the current directory is ignored —
   the script's own metadata wins, matching uv's behavior. Passing
   `--directory` together with a metadata-bearing script is an error, since
   it can only mean a conflict.
3. **Everything else is MEP 3.** Lock/resolve the block's spec into the
   ephemeral root, sync, check `matlab`, activate scoped, run, restore.
   `--with` works unchanged.

A script without a block behaves exactly as in MEP 3.

### Editing commands

Mirroring `uv init/add/remove --script`, the MEP 2 spec-editing commands gain
a `--script <file>` form that edits the metadata block in place instead of a
`mipenv.yaml`:

```
mip env init --script analysis.m             Insert an empty metadata block
mip env add --script analysis.m chebfun      Add deps to the block
mip env remove --script analysis.m chebfun   Remove deps from the block
mip env status --script analysis.m           Show declared vs. cached state
```

Hand-editing the block is equally supported — the commands are a convenience,
not the interface.

### Locking

By default a script is resolved against its channels at run time (cached
until the block changes). For a stronger guarantee, `mip env lock --script
analysis.m` writes a sidecar lockfile `analysis.m.lock` — same JSON format as
`mipenv.lock` — and `mip run` syncs from it when present, with no network
resolution. This matches `uv lock --script`. The sidecar is opt-in: it trades
away single-file-ness, which is the right default for casual sharing but the
wrong one for archival reproducibility (a paper supplement ships both files).

## Backward compatibility

Purely additive. The block is an ordinary comment, inert to MATLAB itself and
to every existing mip command; a metadata-bearing script still runs anywhere
MATLAB runs if its dependencies happen to be on the path. Older mip versions
without this feature simply treat the script as they do today. No change to
package manifests (`mip.yaml`) — packages already have a home for metadata;
this MEP is only about standalone scripts.

## Alternatives

- **TOML content, verbatim PEP 723.** PEP 723 chose TOML because Python
  packaging already speaks TOML. mip speaks YAML everywhere (`mip.yaml`,
  `mipenv.yaml`) and has a YAML parser in-tree; introducing TOML would add a
  parser dependency solely for this block.
- **Block type `script` instead of `mip`.** PEP 723 names the block after the
  content kind (`# /// script`) because multiple Python tools consume it. In
  MATLAB there is no ecosystem-wide standard to be neutral toward; `/// mip`
  says exactly which tool defines the schema.
- **A sidecar spec file** (`analysis.mipenv.yaml` next to the script). Keeps
  parsing trivial but defeats the point: the unit of sharing stops being one
  file. (The opt-in lock sidecar accepts this trade-off deliberately, for a
  use case that already ships multiple files.)
- **A runtime declaration API** (`mip.require('chebfun@1.2.0')` at the top of
  the script). Executable, so it works without `mip run` — but it entangles
  environment setup with execution: the script mutates global state as a side
  effect of running, tools can't read the requirements statically, and
  scoping/restoration is lost.

## Open questions

- **Toolbox requirements.** MATLAB scripts often need toolboxes (checkable
  via `ver`/`license`), which mip does not manage. Is a `toolboxes:` field
  that `mip run` checks (like `matlab`) worth adding, or scope creep?
- **Cache eviction.** Ephemeral environments accumulate under
  `<root>/script-envs/` as metadata evolves. A `mip env clean` (or age-based
  pruning) is probably wanted; here or a follow-up?
- **Version specifier grammar.** Is `>=` on a release name enough, or are
  ranges (`>=R2021a, <R2024a`) needed from day one?
- **Architecture constraints.** A script depending on MEX-bearing packages is
  implicitly per-arch. Should the block be able to say so explicitly, or is
  resolution-time failure clear enough?
- **Block placement.** Leading and trailing placements are both allowed;
  should the spec standardize on one? Top-of-file matches PEP 723 and puts
  requirements where a reader looks first; end-of-file sidesteps the `help`
  collision entirely.
- **Fence spelling.** `% /// mip` follows PEP 723's `# /// script` closely;
  is the parallel worth more than something more MATLAB-idiomatic
  (e.g. `%% mip:`)?
