## Summary

A MEP (Mip Enhancement Proposal) is a design document for a substantial
change to [mip](https://github.com/mip-org/mip). It records the motivation,
the design, the alternatives considered, and the decision, so that design
discussions have a home and their outcomes aren't buried in issue threads.
This MEP describes when a MEP is needed and how one moves from idea to
implementation.

## When is a MEP needed?

Most changes don't need one. Bug fixes, internal refactors, new packages in a
channel, documentation, and small additive features go through ordinary
issues and pull requests on the relevant repository.

A MEP is warranted for changes that are hard to reverse or that other things
are built on top of, such as:

- file formats: the package manifest (`mip.yaml`), installed-package metadata
  (`mip.json`), `.mhl` archives, channel indexes;
- naming and resolution: FQN syntax, bare-name resolution order, version
  selection;
- filesystem layout: the mip root, per-package directory structure,
  persistent state;
- major new subsystems or command groups;
- breaking changes to the CLI;
- the MEP process itself.

When in doubt, open an issue and ask; a maintainer can call for a MEP if the
discussion outgrows the issue.

## Workflow

1. **Float the idea** in a [mip issue](https://github.com/mip-org/mip/issues)
   or a [meps issue](https://github.com/mip-org/meps/issues). This is the
   place to find out whether the idea has legs before investing in a full
   write-up. Prototype code is welcome but not required.
2. **Write the draft.** Copy [`mep-template.md`](../mep-template.md) to
   `meps/mep-XXXX.md`, taking the next unused number (zero-padded to four
   digits in the filename). Open a pull request against
   [mip-org/meps](https://github.com/mip-org/meps).
3. **Merge as Draft.** Review at this stage is for completeness and clarity,
   not acceptance. Once the draft is coherent enough to discuss, it is merged
   with status **Draft**. Merging a draft signals nothing about the outcome.
4. **Discuss and revise.** Discussion happens on the linked issue; revisions
   land as follow-up pull requests to the MEP file.
5. **Decide.** Acceptance requires the agreement of the mip maintainers. The
   status changes to **Accepted** or **Rejected** via a pull request that
   records the rationale (in the MEP itself, typically under a *Resolution*
   section).
6. **Implement.** Once the change lands in mip, the status changes to
   **Implemented**, linking the implementing pull request or release.

After acceptance, small corrections and clarifications are fine as follow-up
PRs. A change to the substance of an accepted design should be a new MEP that
supersedes the old one.

## Statuses

| Status          | Meaning                                                        |
|-----------------|----------------------------------------------------------------|
| **Draft**       | Under discussion; the design may change.                       |
| **Accepted**    | Approved by the maintainers; awaiting implementation.          |
| **Implemented** | The change has landed in mip.                                  |
| **Rejected**    | Considered and declined; kept for the record.                  |
| **Withdrawn**   | Retracted by its author(s).                                    |
| **Superseded**  | Replaced by a later MEP (which it links to).                   |
| **Active**      | A process MEP (like this one) that is in force indefinitely.   |

## Format

MEPs are Markdown files in `meps/`, named `mep-XXXX.md`. Each begins with a
header table (number, title, authors, status, created date, discussion link)
followed by the sections in [`mep-template.md`](../mep-template.md): Summary,
Motivation, Proposal, Backward compatibility, Alternatives, and Open
questions. Sections may be added or dropped where they don't fit; the header
table and Summary are the only hard requirements. The index in the
[README](../README.md) is updated in the same PR as any status change.
