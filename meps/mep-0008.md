## Summary

Give mip first-class **environments**: addressable package roots you create,
activate, and install into by hand — the MATLAB analog of a conda env or a
Python venv. Today mip installs everything into one global root (`~/.mip`);
this MEP adds two more kinds of root — **named** central environments
(`~/.mip/envs/<name>`, conda-style) and **local** project environments
(`./.mip`, venv-style) — and a session pointer (`mip activate` /
`mip deactivate`) that decides which root the session's commands act on.

Package management stays imperative: whatever you `mip install` into the
active env is what is there, the env's contents are the source of truth, and
there is **no lockfile**. This is deliberately the "pip+venv" workflow,
supported as a first-class path rather than a legacy one. A companion MEP
([MEP 9](mep-0009.md)) layers a declarative, lockfile-driven ("uv") workflow
on top of the very same environments; the presence of a lockfile is the only
thing that switches between them.

## Motivation

mip installs everything into one global root, so two projects on the same
machine share one package set and there is no way to give a project — or an
experiment — its own isolated set of packages.

Reproducibility is one reason to want isolation, but not the only one. Often
you just want a clean, throwaway package set for one project without dragging
in a lockfile and a resolver: create an env, install a few packages, work,
and delete it when you're done. That is exactly what `python -m venv` +
`pip install` and `conda create` + `conda install` give their users, and
mip has no equivalent.

Two audiences want two shapes of this. Users coming from conda expect
**named** environments in a central store they can list and activate by name
from anywhere. Users coming from venv expect a **local** environment that
lives beside their code. This MEP supports both with one disambiguation rule.

The earlier [MEP 2](mep-0002.md) also set out to add project environments,
but it coupled them to a lockfile and a separate uv-style spec from the
start, put every operation under an `mip env` subcommand, and mixed venv-style
and uv-style idioms in one command group. It was rejected for that. This MEP
takes the opposite tack: hand-managed environments are defined here on their
own terms; the declarative layer is [MEP 9](mep-0009.md); and the two are
distinguished only by whether a lockfile exists.

## Proposal

### Three kinds of environment

An environment is a mip root — a directory with a `packages/` subtree, cache,
and install state, exactly like today's `~/.mip`. There are three places one
can live, and any of the three can be the **active** env:

| Kind       | Location             | Created by            | Inventoried? |
|------------|----------------------|-----------------------|--------------|
| **global** | `~/.mip`             | bootstrap (today)     | n/a (always) |
| **named**  | `~/.mip/envs/<name>` | `mip env create <name>` | yes (`mip env list`) |
| **local**  | `./.mip`             | `mip env create`      | no (self-managed) |

The global root is the default and behaves exactly as it does today: it is
never bulk-loaded, and packages in it are loaded selectively with `mip load`.
Named and local roots are new. mip keeps an inventory of named envs (it owns
`~/.mip/envs/`); it keeps **no** inventory of local/path envs, which are the
user's to manage like any other directory.

The mechanism already exists: `mip.paths.root()` honors the `MIP_ROOT`
environment variable, so an environment is just a root that `mip activate`
points `MIP_ROOT` at for the session.

### Environment objects — `mip env`

The `mip env` group manages environments as **objects** — nothing more:

```
mip env create [name|path]   Create an empty env
mip env list                 List named envs (readdir of ~/.mip/envs/)
mip env delete <name>        Delete a named env (confirms)
```

- **`mip env create`** — with no argument, creates `./.mip` in the current
  directory; it never walks (see Rules) and errors only if a `mip.lock` is
  present here (that directory belongs to [MEP 9](mep-0009.md)'s declarative
  mode). A **bare name** creates `~/.mip/envs/<name>` in the central store. A
  **path** (anything with a path separator) creates that directory.
- **`mip env list`** — lists the named envs, by reading `~/.mip/envs/`. Path
  envs do not appear; mip does not track them.
- **`mip env delete <name>`** — deletes a named env. This is the **only**
  data-destroying verb in mip and always confirms (`--yes` skips). It takes
  **names only**: given a path it errors ("path envs are self-managed —
  delete the directory yourself"), because mip won't recursively delete an
  arbitrary directory it doesn't own.

Environment arguments are disambiguated **syntactically**, never by guessing:
a bare word is a name (created in and looked up from `~/.mip/envs/` only, with
no fallback to a local path of that name); anything containing a path
separator is a path.

### Activation — session commands

Activation moves a single session-wide pointer:

```
mip activate [name|path]   Point the session at an env
mip deactivate             Point the session back at the global root
mip status                 Show the active env and its health
```

- **`mip activate`** with a name or path is explicit. With **no argument** it
  activates `./.mip` in the current directory, erroring with the one-step fix
  (`mip env create`) if there is none. ([MEP 9](mep-0009.md) refines the
  no-argument case to walk up to the nearest project's `.mip`; without MEP 9
  it is the current directory only.)
- Activation is **exclusive** (one env active at a time) and
  **disk-read-only**: it never installs and never executes package code.
- Whether `mip activate` also **loads** the env's package set onto the path,
  or only moves the pointer, is an open question (below); `--load` /
  `--no-load` exist either way. Regardless, the global root is never
  bulk-loaded.
- **`mip deactivate`** returns the session to the global root and restores the
  prior session state.
- **`mip status`** reports the active env, the resolved project, and health.
  In a hand-managed env the mode is "pip+venv (no lock)"; the richer
  mode/drift reporting arrives with [MEP 9](mep-0009.md). `--check` gives a
  CI-friendly exit code.

### Two ways a command finds its target

This is the rule the whole design turns on, and the two ways are never mixed:

- **Session commands** (`install`, `uninstall`, `update`, `load`, `unload`,
  `list`, `info`, `avail`, `freeze`, …) act on the **active env**, ignoring
  what directory you are in. Session commands **never discover anything on the
  filesystem** — no walking, no sniffing for a nearby `.mip`, ever.
- **Project commands** (introduced in [MEP 9](mep-0009.md): `add`, `lock`,
  `sync`, `run`) act on the nearest `mip.yaml`, walking up from the cwd. They
  ignore activation state.

The asymmetry is deliberate. This is the uv/pip footgun we do **not** import:
only committed, user-owned spec files may be discovered by walking; mutable
env state may never be. So `mip install chebfun` always goes to the env you
activated, never to some `.mip` that happens to be under your cwd.

### The hand-managed (pip+venv) workflow

You manage the environment by hand; mip installs what you tell it.

```
mip install <pkg>            Add a package to the active env
mip uninstall <pkg>          Remove one from the active env
mip update <pkg> | --all     Update packages in the active env
mip install .                Install the local package + its declared deps, editable
mip freeze                   Print the active env as spec lines
```

`install`, `uninstall`, and `update` keep exactly their [documented
behavior](https://github.com/mip-org/mip); the only change is that they now
target the active env (the global root when nothing is activated).

`mip install .` is the `pip install -e .` analog: it installs the local
package in the current directory as an editable install (as `mip install -e .`
does today) into the active env, together with its declared dependencies.

`mip freeze` prints the active env's contents as spec lines — the manual
route to reproducibility without a lock, and the bridge into MEP 9 (its
`mip init --from-env` consumes exactly this).

A typical session:

```matlab
mip env create            % ./.mip
mip activate              % point the session at it
mip install chebfun       % install into it, by hand
mip install .             % ...plus this project's own declared deps, editable
```

Delete `./.mip` and the packages are gone; the env's contents were the only
record of what was in it. To capture that record, `mip freeze` — or adopt the
lockfile workflow of [MEP 9](mep-0009.md).

### Verb discipline

No verb carries two meanings:

| Verbs                     | Meaning                                    |
|---------------------------|--------------------------------------------|
| `add` / `remove`          | edit a spec's dependency list (MEP 9)      |
| `install` / `uninstall`   | mutate env contents                        |
| `create` / `delete`       | make and destroy env objects (`delete` confirms) |
| `load` / `unload`         | toggle a package on/off the session path   |
| `activate` / `deactivate` | move the session pointer                   |

### Unchanged commands

`load`, `unload`, `list`, `info`, `avail`, `init` (package scaffold), `pin`,
`unpin`, `compile`, `test`, `reset`, `bundle`, and the `channel` subcommands
keep their current behavior. They simply act on the active env, which defaults
to the global root — so with nothing activated, they behave exactly as today.

## Backward compatibility

Purely additive. With no env created and none activated, every command
targets the global root exactly as it does today; `mip install`, `mip load`,
`mip list`, and the rest are unchanged in behavior and simply gain an "active
env" notion that defaults to global. The only new surface is the `mip env`
group and `mip activate` / `mip deactivate` / `mip status` / `mip freeze`.

## Alternatives

- **Couple environments to a lockfile from the start (the rejected
  [MEP 2](mep-0002.md)).** This forces the reproducibility machinery — a
  spec, a resolver, a lock — on people who only want an isolated package set,
  and it makes the hand-managed workflow a second-class citizen of the
  lockfile one. Separating the two lets the lock be a clean opt-in
  ([MEP 9](mep-0009.md)) and lets this workflow stand on its own.
- **Only local envs (venv) or only named envs (conda).** Both audiences are
  real. One syntactic disambiguation rule (bare word = name, separator =
  path) supports both without ambiguity.
- **Put the package/session verbs under `mip env` (MEP 2's shape).** That made
  everyday operations second-class. Here `mip env` manages only env *objects*;
  `install`/`activate`/`freeze` are top-level.
- **Let a bare word fall back from a named env to a local path.** Rejected as
  ambiguous. Disambiguate by syntax instead, and keep named and path envs in
  strictly separate namespaces (mip inventories the former, not the latter).

## Open questions

- **Does `mip activate` load by default?** Attach-semantics (activation loads
  the env's package set, and load failures surface as mip errors with
  remedies) versus pointer-only (nothing is on the path unless you `mip load`
  or `mip run` it — the uniform rule mip uses today). One flag settles it;
  `--load` / `--no-load` exist regardless. To be decided by dogfooding.
- **`mip env` or `mip venv`?** `env` reads as conda's noun and covers named
  central envs, which `venv` does not; `venv` is more familiar to the Python
  crowd but misdescribes the named case.
- **Silent global target.** When nothing is activated, should `mip install`
  warn that it is writing to the global root, or is silent-global the right
  default (it is today's behavior)?
- **`mip freeze` output.** Bare spec lines, or a ready-to-commit `mip.yaml`
  that flows directly into [MEP 9](mep-0009.md)?
