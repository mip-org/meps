# MEP 8: Environments (`mip env`)

|                |                                                              |
|----------------|--------------------------------------------------------------|
| **MEP**        | 8                                                            |
| **Title**      | Environments (`mip env`)                                     |
| **Author(s)**  | Dan Fortunato, Jeremy Magland                                |
| **Status**     | Draft                                                        |
| **Created**    | 2026-07-10                                                   |
| **Discussion** | [mip-org/meps#8](https://github.com/mip-org/meps/issues/8)   |

## Summary

Give mip first-class **environments**: addressable package roots you create,
activate, and install into by hand — the MATLAB analog of a conda env or a
Python venv. Today mip installs everything into one global root (the
directory mip itself is installed into, by default `<userpath>/mip`, e.g.
`~/Documents/MATLAB/mip`);
this MEP adds two more kinds of root — **named** central environments
(`<root>/envs/<name>`, conda-style) and **local** project environments
(`./.mip`, venv-style) — and a session pointer (`mip env activate` /
`mip env deactivate`) that decides which root the session's commands act on.

Package management stays imperative: whatever you `mip install` into the
active env is what is there, the env's contents are the source of truth, and
there is **no lockfile**. This is deliberately the "pip+venv" workflow,
supported as a first-class path rather than a legacy one. A companion MEP
([MEP 9](mep-0009.md)) layers a declarative, lockfile-driven ("uv") workflow
on top of the very same environments; the presence of a lockfile is the only
thing that switches between them.

## Motivation

mip installs everything into one global root, so two projects on the same
machine share one package set and there is no way to give a project — or an
experiment — its own isolated set of packages.

Reproducibility is one reason to want isolation, but not the only one. Often
you just want a clean, throwaway package set for one project without dragging
in a lockfile and a resolver: create an env, install a few packages, work,
and delete it when you're done. That is exactly what `python -m venv` +
`pip install` and `conda create` + `conda install` give their users, and
mip has no equivalent.

Two audiences want two shapes of this. Users coming from conda expect
**named** environments in a central store they can list and activate by name
from anywhere. Users coming from venv expect a **local** environment that
lives beside their code. This MEP supports both with one disambiguation rule.

The earlier [MEP 2](mep-0002.md) also set out to add project environments,
but it coupled them to a lockfile and a separate uv-style spec from the
start, put every operation under an `mip env` subcommand, and mixed venv-style
and uv-style idioms in one command group. It was rejected for that. This MEP
takes the opposite tack: hand-managed environments are defined here on their
own terms; the declarative layer is [MEP 9](mep-0009.md); and the two are
distinguished only by whether a lockfile exists.

## Proposal

### Command reference

Target column: **by session** = the active env, regardless of directory.
(**By project** — the nearest `mip.yaml` walking up from cwd — arrives with
[MEP 9](mep-0009.md)'s project commands; nothing in this MEP walks.)

| Command | What it does | Touches | In CLI? |
|---|---|---|---|
| `mip env create [name\|path]` | Empty env: no arg → `./.mip` in cwd, no walking; bare name → central store `<baseline root>/envs/<name>`; path → that dir | new root, as specified | y |
| `mip env list` | List named envs (readdir of `<baseline root>/envs/`); marks the active one | nothing | y |
| `mip env delete <name>` | Delete a named env — the only data-destroying verb; confirms (`--yes` to skip). Names only: given a path, it errors ("path envs are self-managed — delete the directory yourself") | root in `envs/` | y |
| `mip env activate [name\|path]` (alias `mip activate`) | Point the session at an env (no arg → `./.mip` in the current directory). Pointer-only by default; `--load` also loads the env's package set | session pointer + MATLAB path swap | n |
| `mip env deactivate` (alias `mip deactivate`) | Point the session back at the baseline root; restores the prior session state | session pointer + MATLAB path swap | n |
| `mip env` | Show the active env (or that none is active) | nothing | n |

Activation is session state inside MATLAB, so `activate`/`deactivate` have no
meaning in the shell CLI ([MEP 6](mep-0006.md)); there, a specific root is
targeted by setting the `MIP_ROOT` environment variable.

### Three kinds of environment

An environment is a mip root — a directory with a `packages/` subtree, cache,
and install state, exactly like today's global root. There are three places one
can live, and any of the three can be the **active** env:

| Kind       | Location                     | Created by              | Inventoried? |
|------------|------------------------------|-------------------------|--------------|
| **global** | where mip is installed (default `<userpath>/mip`) | bootstrap (today) | n/a (always) |
| **named**  | `<baseline root>/envs/<name>` | `mip env create <name>` | yes (`mip env list`) |
| **local**  | `./.mip`                     | `mip env create`        | no (self-managed) |

The global root is the default and behaves exactly as it does today: it is
never bulk-loaded, and packages in it are loaded selectively with `mip load`.
Named and local roots are new. mip keeps an inventory of named envs (it owns
the `envs/` store); it keeps **no** inventory of local/path envs, which are
the user's to manage like any other directory.

**The baseline root.** The *baseline root* is the root the session uses when
no environment is active: the value of `MIP_ROOT` at session start if set
externally, otherwise the root derived from mip's own install location
(normally `<userpath>/mip`, e.g. `~/Documents/MATLAB/mip`). Activation never
moves the baseline — `mip deactivate`
always returns to it — and the named-env store lives at
`<baseline root>/envs/`. Anchoring the store to the baseline (rather than a
hard-coded location) means a test suite or CI job that sets `MIP_ROOT` keeps
its named envs inside that root too. Named env operations (`create <name>`,
`list`, `delete <name>`, `activate <name>`) always resolve against the
baseline store, even while some other env is active.

**Anatomy of an env.** `mip env create` materializes exactly one thing: an
empty `packages/` subtree. That subtree is also what makes the directory an
environment — it is the very signal `mip.paths.root()` already keys on (an
existing directory containing a `packages/` subdirectory), so an env is
structurally identical to the global root rather than a new kind of directory.
There is no separate marker file: a directory *is* an env exactly when it
holds a `packages/` subtree. `mip activate` refuses a directory without one
("not a mip environment — create one with `mip env create`"), and
`mip env delete` — which takes a name, never a directory, and so resolves it
to a directory in the `envs/` store — requires a `packages/` subtree on that
resolved directory before it will recursively remove it. Everything else in a
root (cache, trash, state files) is created lazily by the existing machinery,
exactly as for the global root today.

The mechanism already exists: `mip.paths.root()` honors the `MIP_ROOT`
environment variable, so an environment is just a root that `mip activate`
points `MIP_ROOT` at for the session (via `setenv`, after saving the prior
value). Child processes spawned from the session — `system()` calls, a
`matlab -batch` — inherit the active env, which is deliberate.

### Environment objects — `mip env`

The `mip env` group manages environments as **objects** — nothing more:

```
mip env create [name|path]   Create an empty env
mip env list                 List named envs
mip env delete <name>        Delete a named env (confirms)
```

- **`mip env create`** — with no argument, creates `./.mip` in the current
  directory; it never walks (see the targeting rule below). A **bare name**
  creates `<baseline root>/envs/<name>` in the central store — always the
  baseline store, even while another env is active. A **path** (anything
  containing a path separator) creates that directory. On success it prints
  the created path and an activation hint. Creation is **strict**:
  - target is already an env (a `packages/` subtree present) → error
    ("environment already exists");
  - target is an existing non-empty directory that is not an env → error
    (mip will not adopt arbitrary directories);
  - target is an empty directory or does not exist → created;
  - the no-argument form additionally errors if a `mip.lock` is present in
    the current directory (that directory belongs to [MEP 9](mep-0009.md)'s
    declarative mode, where `.mip/` is derived by `mip project sync`).

  A local `./.mip` env should normally be added to the project's
  `.gitignore`; mip does not edit user files, so this is a documentation
  recommendation, not runtime behavior.
- **`mip env list`** — lists the named envs by reading
  `<baseline root>/envs/` (entries without a `packages/` subtree are ignored),
  marking the active one. Path envs do not appear; mip does not track them.
- **`mip env delete <name>`** — deletes a named env. This is the **only**
  data-destroying verb in mip and always confirms (`--yes` skips; the
  `MIP_CONFIRM` environment variable is honored for non-interactive use, as
  in local install's auto-generate prompt). It takes **names only**: given a
  path it errors ("path envs are self-managed — delete the directory
  yourself"), because mip won't recursively delete an arbitrary directory it
  doesn't own. It refuses to delete the **active** env ("run `mip deactivate`
  first") and refuses a directory that has no `packages/` subtree (i.e. isn't a
  mip root).

Environment arguments are disambiguated **syntactically**, never by guessing:
a bare word is a name (created in and looked up from the baseline `envs/`
store only, with no fallback to a local path of that name); anything
containing a path separator (`/`, or `\` on Windows) is a path. Names are
validated with the same rule as package-name components (`mip.name.is_valid`:
letters, digits, hyphens, underscores; starts and ends alphanumeric; no
dots), and are compared exactly as typed — no case or separator
normalization.

### Activation — session commands

Activation moves a single session-wide pointer and swaps the session's load
state with it:

```
mip env activate [name|path] [--load]   Point the session at an env
mip env deactivate                      Point the session back at the baseline root
mip env                                 Show the active env
mip activate / mip deactivate           Aliases for the above
```

**`mip activate`** with a name resolves against the baseline `envs/` store;
with a path, against that directory (stored and displayed as an absolute
path); with **no argument** it activates `./.mip` in the current directory.
In every form the target must be an env — an existing directory with a
`packages/` subtree — and anything else errors with the one-step fix
(`mip env create ...`). ([MEP 9](mep-0009.md) refines the no-argument case to
walk up to the nearest project's `.mip`; without MEP 9 it is the current
directory only.)

Activation is **exclusive** (one env active at a time) and **disk-read-only**
with respect to the env: it never installs and never executes package code
(loading and unloading in mip are pure `addpath`/`rmpath`). The steps:

1. If an env is already active: if it is the *same* env, print "already
   active" and stop (though `--load` still performs its load pass — the
   natural way to say "load everything in this env now"); otherwise run the
   full `mip deactivate` first, then continue.
2. Save the session state: the current `MIP_ROOT` value and the
   loaded / directly-loaded / sticky lists.
3. Unload every loaded package **including sticky ones** via the normal
   unload machinery — path entries removed, MEX binaries cleared (so no
   DLL from another root stays locked on Windows). Only the **running
   mip** stays: `gh/mip-org/core/mip`, plus the loaded package actually
   providing the running `mip` code when that is a different one (e.g. a
   preview build of mip loaded from another channel over the released
   mip). Implicit unloads — this swap, `mip deactivate`,
   `mip unload --all --force`, `mip reset`, and transitive-dependency
   pruning — never pull the running mip off the path; an explicit
   `mip unload` of a preview build remains allowed (that is how a preview
   is exited).
4. `setenv('MIP_ROOT', <env>)`.
5. Reset the session load state to the usual baseline (mip always loaded and
   sticky).
6. With **`--load`**: load each of the env's *directly installed* packages as
   a direct load — dependencies load transitively, exactly as if the user had
   run `mip load` on each. Loading is best-effort: the pointer swap has
   already completed, each failure prints the mip error and remedy, and the
   command ends with a summary (loaded N, failed M). The env stays active
   regardless.

Without `--load` (the default), activation is pointer-only: nothing is on the
path until you `mip load` it — the uniform rule mip uses today. The global
root is never bulk-loaded in any case.

**`mip deactivate`** restores the prior session state:

1. If no env is active: print a message and do nothing.
2. Unload all loaded packages except the running mip (the same rule as
   activation step 3). This works even if the env directory was deleted
   out from under the session — path entries are swept rather than
   resolved from disk.
3. Restore `MIP_ROOT` to its saved value (which may be an externally set
   custom root, or unset).
4. Restore the saved package set: each formerly loaded package goes back on
   the path with its prior direct/sticky flags, and in its prior **load
   order** — packages are re-added in the sequence they were originally
   loaded, so path precedence (which copy of a shadowed function wins) is
   exactly what it was before activation. Restoration is best-effort with
   warnings (mirroring `--load`) — e.g. a package uninstalled meanwhile by
   another MATLAB session warns rather than aborts.

Because `mip activate` auto-deactivates first, the saved state is always the
baseline session's — there is no activation stack, and an env's load state is
not remembered across activations (each activation starts cold).

**`mip env`** reports the active env — the name (for named envs) or absolute
path, e.g. `active environment: scratch (~/Documents/MATLAB/mip/envs/scratch)`.
If none is
active it reports the baseline root and how to create and activate an env.
`mip info` (with no argument) also reports the active env. Richer
status/drift reporting (and its CI-friendly `--check` exit code) arrives with
[MEP 9](mep-0009.md)'s `mip project status`.

Two MATLAB sessions may activate the same env concurrently: on-disk state is
shared (exactly as today's global root is shared across sessions) and load
state is per-session.

### Showing the target

Because activation is session state with no shell prompt to reflect it,
commands that mutate the env — `install`, `uninstall`, `update` — and
`mip list` print one leading line **when an env is active**:

```
environment: scratch (~/Documents/MATLAB/mip/envs/scratch)
```

(for a path env, the path alone). When nothing is active, output is exactly
today's — the global root is a first-class default target, not a mistake, so
there is no "installing into global root" warning.

### How a command finds its target

This is the rule the whole design turns on:

- **Session commands** (`install`, `uninstall`, `update`, `load`, `unload`,
  `list`, `info`, `avail`, …) act on the **active env**, ignoring what
  directory you are in. Session commands **never discover anything on the
  filesystem** — no walking, no sniffing for a nearby `.mip`, ever.
- **Project commands** (introduced in [MEP 9](mep-0009.md)'s `mip project`
  group: `add`, `lock`, `sync`, `run`, …) act on the nearest `mip.yaml`,
  walking up from the cwd. They ignore activation state.

The asymmetry is deliberate. This is the uv/pip footgun we do **not** import:
only committed, user-owned spec files may be discovered by walking; mutable
env state may never be. So `mip install chebfun` always goes to the env you
activated, never to some `.mip` that happens to be under your cwd.

(`mip env create` and no-argument `mip activate` do refer to the current
directory — but explicitly and without walking: they act on `./.mip` exactly,
as stated on the tin.)

### Per-root state

Every root is fully self-contained, as the global root is today: package
tree, `directly_installed.txt`, pins, channel subscriptions
(`packages/channels.txt`), and the channel index cache are all per-root. A
fresh env therefore starts with no subscribed channels, no pins, and a cold
index cache; `mip channel add` inside an env affects that env only. A shared
download/index cache across envs is possible future work, not part of this
MEP.

### The hand-managed (pip+venv) workflow

You manage the environment by hand; mip installs what you tell it.

```
mip install <pkg>            Add a package to the active env
mip uninstall <pkg>          Remove one from the active env
mip update <pkg> | --all     Update packages in the active env
mip install -e .             Install the local package editable, with its declared deps
```

`install`, `uninstall`, and `update` keep exactly their [documented
behavior](https://github.com/mip-org/mip); the only change is that they now
target the active env (the global root when nothing is activated).

**Local installs gain auto-deps.** One general improvement ships with this
MEP: local installs (copy and editable alike, in any root) now auto-install
missing channel dependencies declared in `mip.yaml`, instead of erroring with
`mip:dependencyNotFound`
([mip-org/mip#161](https://github.com/mip-org/mip/issues/161)). As with remote
installs, such dependencies are installed transitively (not marked directly
installed). This aligns the local path with the remote one, which already runs
`installMissingDeps` after download; the same fix covers `mip update` of a
local package, which errors today for the same reason. It also makes
`mip install -e .` the `pip install -e .` analog: the local package goes in
editable, together with its declared dependencies. Defaults are otherwise
unchanged — a local path without `-e` remains a copy install.

A typical session:

```matlab
mip env create            % ./.mip
mip activate              % point the session at it
mip install chebfun       % install into it, by hand
mip install -e .          % ...plus this project's own declared deps, editable
```

Delete `./.mip` and the packages are gone; the env's contents were the only
record of what was in it. Capturing that record reproducibly is the lockfile
workflow of [MEP 9](mep-0009.md).

### mip itself and environments

mip always runs from the root it was bootstrapped into — activating an env
never relocates mip, and envs are usable without containing it. The
`gh/mip-org/core/mip` identity protections (always loaded and sticky, cannot
be unloaded, survives `unload --all --force`, never pruned) are FQN-based and
session-wide, so they hold in every env.

The **self-** flows key on *which root is active*: self-uninstall (the flow
that tears down the entire root) and self-update trigger **only when the
active root is the root mip actually runs from**. While an env is active,
`gh/mip-org/core/mip` is an ordinary package there — `mip install
mip-org/core/mip` installs an (inert) copy into the env, `mip uninstall mip`
removes that copy or reports "not installed", and `mip update mip` updates
it. The running mip is untouched, and `mip load mip` keeps its current
"always loaded" no-op, so an env copy can never shadow the running one. This
guarantees the self-uninstall flow can never delete an env root out from
under the user.

The running mip need not be `gh/mip-org/core/mip`: a preview build of mip
(e.g. `gh/mip-org/labs/mip`) can be installed and loaded over the released
one, and the loaded package providing the running `mip` code inherits the
same treatment from implicit unloads — activation's swap, deactivation,
`unload --all --force`, reset, and pruning all spare it, so a loaded
preview survives `mip activate` / `mip deactivate`. Unlike the core
identity it can still be unloaded explicitly, which is how the preview is
exited.

### Verb discipline

No verb carries two meanings:

| Verbs                     | Meaning                                    |
|---------------------------|--------------------------------------------|
| `add` / `remove`          | edit a spec's dependency list (MEP 9)      |
| `install` / `uninstall`   | mutate env contents                        |
| `create` / `delete`       | make and destroy env objects (`delete` confirms) |
| `load` / `unload`         | toggle a package on/off the session path   |
| `activate` / `deactivate` | move the session pointer                   |

### Unchanged commands

`load`, `unload`, `list`, `info`, `avail`, `init` (package scaffold), `pin`,
`unpin`, `compile`, `test`, `reset`, `bundle`, and the `channel` subcommands
keep their current behavior. They simply act on the active env, which defaults
to the global root — so with nothing activated, they behave exactly as today.

## Backward compatibility

Additive, with one deliberate behavior change. With no env created and none
activated, every command targets the global root and prints exactly what it
does today; the new "environment:" output line appears only while an env is
active. The new surface is the `mip env` group and `mip activate` /
`mip deactivate`.

The one behavior change: local installs now auto-install missing channel
dependencies where they previously raised `mip:dependencyNotFound`. An error
case becomes a success case; no working invocation changes meaning.

## Deferred

- **`mip freeze`** (print the active env as spec lines) is cut from this MEP.
  Capturing an env's contents belongs with the reproducibility layer;
  [MEP 9](mep-0009.md)'s `mip project init --from-env` can read the active env
  directly rather than consuming freeze output. A standalone freeze can be
  revisited later if a need appears.
- **A one-shot `--env <name|path>` override on session commands** (e.g.
  `mip install --env scratch chebfun`). Scripts and CI can already target a
  root via the `MIP_ROOT` environment variable; the shell-CLI story belongs
  to [MEP 6](mep-0006.md).
- **A shared download/index cache across envs** (per-root caches are correct,
  just cold on first use).

## Alternatives

- **Couple environments to a lockfile from the start (the rejected
  [MEP 2](mep-0002.md)).** This forces the reproducibility machinery — a
  spec, a resolver, a lock — on people who only want an isolated package set,
  and it makes the hand-managed workflow a second-class citizen of the
  lockfile one. Separating the two lets the lock be a clean opt-in
  ([MEP 9](mep-0009.md)) and lets this workflow stand on its own.
- **Only local envs (venv) or only named envs (conda).** Both audiences are
  real. One syntactic disambiguation rule (bare word = name, separator =
  path) supports both without ambiguity.
- **Put the package/session verbs under `mip env` (MEP 2's shape).** That made
  everyday operations second-class. Here `mip env` manages only env *objects*;
  `install`/`activate` are top-level.
- **Let a bare word fall back from a named env to a local path.** Rejected as
  ambiguous. Disambiguate by syntax instead, and keep named and path envs in
  strictly separate namespaces (mip inventories the former, not the latter).
- **Leave loaded packages in place across `mip activate`.** Rejected: load
  state is one session-global FQN list, so a package loaded from another root
  would collide with the same FQN in the new env (`mip load` would no-op
  against the wrong root's copy). The full swap keeps every root's view
  self-consistent, and since loading is `addpath`-only, the swap executes no
  package code.
- **Auto-edit `.gitignore` on local `mip env create`.** Rejected: mip does
  not edit user-owned files. The recommendation lives in documentation.
- **A dedicated `mip-env.json` marker file to identify envs.** An earlier draft
  wrote a marker (format version, creation time, creating mip version) that
  `activate` and `delete` required. Dropped as redundant: `mip.paths.root()`
  already treats any directory with a `packages/` subtree as a root, and every
  env is just a root, so `packages/` is the natural env signal — a marker would
  make envs asymmetric with the global root (which has none) and add a file
  nothing reads. A format-version anchor, if ever needed, belongs to *every*
  root as uniform per-root state under `packages/`, not to an env-only sibling
  file.
- **`mip venv` instead of `mip env`.** `env` covers named central envs, which
  `venv` misdescribes, and reads as conda's noun.

## Resolved questions

Decisions reached in review, recorded for the discussion thread:

- **Does `mip activate` load by default?** No — pointer-only, matching mip's
  uniform "nothing bulk-loads" rule; `--load` opts in (direct installs as
  direct loads, dependencies transitive, best-effort with a summary).
- **What happens to already-loaded packages on activate?** Full swap:
  activate unloads everything (sticky included, the running mip excepted)
  after saving the session state; deactivate unloads the env's packages and
  restores the saved state. No activation stack; auto-deactivate before
  activating another env.
- **A loaded preview build of mip.** The loaded package that provides the
  running `mip` code — normally `gh/mip-org/core/mip`, but a shadowing
  preview build (e.g. `gh/mip-org/labs/mip`) when one is loaded — is spared
  by every implicit unload (activation's swap, deactivate,
  `unload --all --force`, reset, pruning), so a preview survives
  activation. Explicit `mip unload` of a preview remains allowed; that is
  the exit ramp back to the released mip.
- **Pointer mechanism.** `setenv('MIP_ROOT', ...)` after saving the prior
  value; deactivate restores it. Child processes inherit the active env.
- **Where the named-env store lives.** `<baseline root>/envs/`, where the
  baseline is the session's no-env root (external `MIP_ROOT` respected,
  activation ignored) — so `MIP_ROOT`-isolated test runs stay isolated.
- **What marks a directory as an env.** Nothing beyond a `packages/` subtree —
  the same signal `mip.paths.root()` already uses for any root. `mip env
  create` materializes just that subtree; `activate` and `delete` both require
  it. There is no separate marker file (an earlier draft's `mip-env.json` was
  dropped as redundant with `packages/`).
- **mip itself.** Self-uninstall/self-update trigger only when the active
  root is the root mip runs from; in an env, `gh/mip-org/core/mip` is an
  ordinary (inert) package. Identity protections stay FQN-based.
- **`mip env` or `mip venv`?** `env`, with `mip env activate` /
  `mip env deactivate` as the primary spelling and top-level `mip activate` /
  `mip deactivate` as aliases.
- **Silent global target.** Yes — unchanged, silent. The visibility
  mechanism is the "environment:" line printed when an env *is* active.
- **`mip freeze` output.** Moot — freeze is deferred out of this MEP (see
  Deferred).
- **Per-root state.** Fully per-root (channels, pins, caches,
  `directly_installed.txt`); no inheritance at create time.
- **Edge rules.** Strict create (error on existing env or non-empty non-env
  dir; `mip.lock` guard for the no-arg form); delete refuses the active env
  and always confirms (`--yes`, `MIP_CONFIRM`); re-activating the active env
  is a no-op message (`--load` still loads); env names validate with
  `mip.name.is_valid` and compare exactly.
