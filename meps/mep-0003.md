## Summary

Add `mip run` — the mip analog of `uv run`. It runs a MATLAB script or
expression *inside* a project environment ([MEP 2](mep-0002.md)) without the
user managing that environment by hand: it makes sure the environment is
locked and synced, activates it for the duration of the run only, executes
the code, and then restores the session to its previous state. Where
`mip env activate` changes the session until further notice, `mip run` is
scoped: after it returns, the MATLAB path and mip root are exactly what they
were before.

```matlab
mip run analysis.m                      % sync env in cwd, run script inside it
mip run analysis.m --directory proj     % use proj/mipenv.yaml
mip run "results = solve(42)"           % run an expression instead of a script
mip run analysis.m --with chebfun       % add a package for this run only
```

From the shell (and from tools like calkit), the same thing is one `-batch`
invocation:

```sh
matlab -batch "mip run analysis.m --directory proj"
```

## Motivation

MEP 2 gives a project a declared, locked, reproducible package set, but using
it is manual and stateful: `mip env sync`, `mip env activate`, run your code,
`mip env deactivate` — and if you forget the last step, the rest of your
session silently runs against the project root. This is the same gap `uv`
closes with `uv run`: `uv sync` + activate + run + deactivate, collapsed into
one command that leaves no state behind.

Concretely:

- **Reproducible entry points.** "Run this script against exactly the locked
  packages" becomes a single command that is safe to put in a README, a
  Makefile, a CI job, or a calkit pipeline stage.
- **No leaked session state.** MEP 2 notes that activating a second
  environment in one session mixes paths. A scoped run–restore cycle avoids
  the problem for the common case: you don't stay in the environment at all.
- **Simpler tool integration.** calkit's "run a command in the env" hook
  (`calkit xenv`, analog of `uv run`) currently needs to compose
  `activate` + command itself. With `mip run` it maps one-to-one.

## Proposal

### Command

```
mip run <target> [--directory <dir>] [--with <pkg> ...] [--frozen] [--no-sync]
```

`<target>` is one of:

- a path to a `.m` script — executed with `run`, from the directory it lives
  in (MATLAB's usual script semantics);
- a MATLAB expression in quotes — executed with `evalin` in the base
  workspace, so assignments like `mip run "r = f(3)"` leave `r` behind for
  interactive use.

Flags:

- `--directory <dir>` — the project directory containing `mipenv.yaml` /
  `mipenv.lock`. Defaults to the current folder, matching MEP 2's discovery
  rule (no walking up the tree).
- `--with <pkg>` — additionally install and load `<pkg>` for this run only,
  without touching `mipenv.yaml` or `mipenv.lock`. Repeatable. Accepts the
  same package syntax as `mip install` (bare name, `@version`, FQN).
- `--frozen` — require `mipenv.lock` to exist and be up to date with
  `mipenv.yaml`; error rather than re-lock. For CI, where an implicit
  re-resolve should be a failure, not a convenience.
- `--no-sync` — skip the sync step and run against whatever is already
  installed in the project root. Errors if a locked package is missing.

### Behavior

A run proceeds in four steps, the first two identical to the corresponding
`mip env` commands:

1. **Lock.** If `mipenv.lock` is missing, or `mipenv.yaml` has changed since
   it was written, re-lock (as `mip env lock`). With `--frozen`, error
   instead.
2. **Sync.** Install anything in the lock that is missing from the project
   root `<project>/.mip` (as `mip env sync`), plus any `--with` packages.
3. **Activate, scoped.** Point `MIP_ROOT` at the project root and load the
   environment's direct packages and the `--with` packages — the same effect
   as `mip env activate`, but guarded (via `onCleanup`) rather than for the
   rest of the session.
4. **Run and restore.** Execute the target. On return — normal or error —
   restore the previous `MIP_ROOT`, MATLAB path, and mip load state, then
   rethrow any error. The base workspace is deliberately *not* restored:
   variables the target creates are its output.

Steps 3–4 reuse MEP 2's key mechanism: `mip.env.with_root` already scopes
`MIP_ROOT` with `onCleanup`; `mip run` extends the same guard to the path and
load state. Like `mip env`, the feature is additive — a new dispatch case in
`mip.m` and code in `+mip/+env/` (or a sibling `+mip/+run/`).

### From the shell

mip lives inside MATLAB, so the true analog of typing `uv run …` at a shell
prompt is:

```sh
matlab -batch "mip run analysis.m --directory proj"
```

Here the scoping is trivial (the session ends anyway), and what `mip run`
contributes is the lock/sync/load preamble in one command. This replaces the
two-part `mip env sync` + `mip env activate; <command>` recipe in MEP 2's
calkit integration: calkit's `xenv` runner for `kind: mip` becomes a single
`mip run` wrapping of the user's command.

### What this MEP is not

`uv run --script script.py` reads dependency metadata embedded in the script
itself (PEP 723) and builds an ephemeral environment from it, with no project
directory at all. That is a natural follow-up — `mip run` is the execution
vehicle it would plug into — but it is deliberately out of scope here and
will be proposed separately.

## Backward compatibility

Purely additive: `run` is not currently a mip command, and no existing
behavior changes. Requires MEP 2's machinery to exist.

## Alternatives

- **Do nothing; compose `env sync`/`activate`/`deactivate` by hand.** Works,
  but stateful and easy to get wrong (a forgotten `deactivate`, an error
  between activate and deactivate); tools would each reinvent the same
  wrapper.
- **Make `activate` scoped instead.** An activation that only lasts for the
  current function scope isn't useful interactively, where activate/deactivate
  are typed at the prompt. The two commands serve different modes: `activate`
  for interactive sessions, `run` for entry points.
- **A shell-side wrapper (`mipx`).** A standalone executable could mimic
  `uv run` ergonomics from the terminal, but mip has no shell presence today,
  and `matlab -batch "mip run …"` gets the same effect without introducing
  one.
- **Naming: `mip exec`.** `run` matches both `uv run` and MATLAB's own `run`
  function, and mip has no conflicting command.

## Open questions

- **Functions with arguments.** Scripts and expressions cover most cases
  (`mip run "f(3)"` handles functions), but is a first-class form like
  `mip run f 3` worth the parsing ambiguity?
- **Automatic re-lock.** Step 1 re-locks when the spec is newer than the
  lock, following `uv run`. Should the default instead be `--frozen`-like
  strictness, given MATLAB users may be less accustomed to implicit
  resolution?
- **Where `--with` packages install.** Into the project root (simple, but
  accumulates packages the lock doesn't mention — should `mip env sync`
  prune them?) or a separate ephemeral overlay root (clean, but new
  machinery)?
- **State beyond the path.** Restoring path + `MIP_ROOT` + load state covers
  mip-managed state, but a run can still leak `javaaddpath` entries, loaded
  MEX files, or globals. Is best-effort restoration acceptable, or should the
  docs steer heavy isolation cases toward `matlab -batch`?
- **Parallel pools.** Workers in a `parpool` do not inherit the temporarily
  modified path automatically. Document, or address?
