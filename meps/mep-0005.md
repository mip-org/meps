## Summary

MLTBX is MathWorks' toolbox packaging format — the unit of distribution for
the Add-On Explorer, much of File Exchange, and click-to-install sharing.
This MEP makes mip interoperate with it at three points:

1. **Install from an MLTBX.** `mip install ./pkg.mltbx` (or a URL ending in
   `.mltbx`) unpacks the toolbox into mip's package tree, translating its
   manifest into a synthesized `mip.json`, so `load`/`unload`/`uninstall`/
   pruning work like for any other package.
2. **Build from an MLTBX.** Channel recipes can name an MLTBX as their
   *upstream source*; the channel build unpacks it and packs an ordinary
   `.mhl`. The MLTBX is a build input, never a payload.
3. **Export to an MLTBX.** `mip bundle --toolbox` produces a distributable
   `.mltbx` from a mip package, the mirror image of the `.mhl` it produces
   today — so mip package authors can reach users who don't have mip.

In all three, MLTBX is a *boundary* format between mip's world and the
MathWorks ecosystem. Internally nothing changes: `.mhl` remains the only
wire format channels serve, and the installed form (`mip.json` + extracted
source) remains the only thing mip manages.

Because mip is itself a mip package, (3) also answers a request from the
announcement discussion: `mip bundle --toolbox` applied to mip produces a
click-to-install `mip.mltbx`, without making MLTBX the primary installation
route.

## Motivation

Feedback after mip's announcement was that MLTBX is how people already share
MATLAB utilities: recipients open the file, confirm, and the toolbox appears
in the Add-On Explorer with version information. That world and mip's world
currently don't touch, which costs both directions:

- **MLTBX in hand, no mip lifecycle.** A user given an MLTBX today installs
  it through the Add-On manager, outside mip: no `mip load`/`unload`, no
  uninstall pruning, invisible to `mip list`.
- **MLTBX-only upstreams.** Some packages — many File Exchange entries — are
  distributed *only* as MLTBX. A channel maintainer wanting to package one
  has no supported way to ingest it.
- **mip authors can't reach non-mip users.** A package developed with mip
  (`mip.yaml`, `mip bundle`) has no path to the Add-On Explorer audience
  short of hand-running MATLAB's toolbox packager with duplicated metadata.

The same feedback cautioned (from experience) against making MLTBX the
*primary* form of anything: it is convenient for distribution and poor for
development. This MEP follows that line — MLTBX at the edges, mip formats at
the core.

## Proposal

### Background: what an MLTBX is

An `.mltbx` file is a zip container (OPC) holding the toolbox files plus a
metadata manifest: name, version, author, summary, a GUID identifying the
toolbox across versions, the folders to add to the path, and optionally
"required add-ons" and screenshots/icons. MATLAB's Add-On manager installs it
by copying the files and permanently editing the path.

### 1. Installing from an MLTBX

```
mip install ./pkg.mltbx                     % file on disk
mip install https://example.com/pkg.mltbx   % direct URL
```

Detection is by extension, within the *existing* source categories: a local
file installs as `local/<name>`, a URL as `web/<name>` — no new FQN category
and no resolver changes. mip unpacks the container, reads the manifest, and
synthesizes `mip.json`:

- **name, version, description** — from the manifest. A missing or
  non-semver version is preserved as-is (mip already tolerates branch-like
  versions).
- **paths** — from the manifest's path entries.
- **dependencies** — the manifest's "required add-ons" are Add-On Explorer
  references (GUIDs), not mip packages; they do not map. They are reported
  as a warning and otherwise ignored.

After that, the package is indistinguishable from any other local/web
install: loadable, unloadable, uninstallable, listed, pruned. Like other
non-channel installs it has no upstream to check, so `mip update` means
handing mip a newer file.

### 2. MLTBX as a channel build source

Channel recipes name an upstream (`repository:`) that the build fetches.
This MEP allows that upstream to be an MLTBX: the build unpacks it (as in
§1, at build time), applies the recipe's own `mip.yaml` — which, as for any
channel package, overrides name/version/paths and supplies real mip
dependencies — and packs a **normal `.mhl`** of the extracted contents.

Users installing such a package go through the ordinary channel flow and
never encounter the MLTBX. Notably, `.mhl` archives do **not** embed intact
`.mltbx` files — see *What this MEP is not*.

Channels also do not serve MLTBX in the other direction: `.mhl` remains the
only format mip installs from a channel. It is per-architecture (prebuilt
MEX selected for the user's platform) and embeds resolved, mip-native
metadata that the channel index and resolver depend on before install —
properties an MLTBX does not have. A channel's CI *may*, however, use §3's
export to attach a `.mltbx` to its GitHub Releases alongside each `.mhl`, as
a convenience download for non-mip users; mip itself never installs from it.

### 3. Exporting with `mip bundle --toolbox`

`mip bundle ./path` today produces a `.mhl` (mip's wheel). With `--toolbox`
it additionally (or instead, `--toolbox-only`) produces
`<name>-<version>.mltbx`, mapping `mip.yaml` onto the toolbox manifest. (The
flag is named for what MathWorks calls the artifact — a toolbox — rather
than its file extension.)

- **name, version, summary, author** — from `mip.yaml`.
- **path folders** — from `mip.yaml`'s `paths`.
- **GUID** — derived deterministically from the package's FQN (e.g. a UUID5
  hash), so successive versions of the same package share a GUID and the
  Add-On Explorer treats them as upgrades of one toolbox rather than
  siblings.
- **dependencies** — a `.mhl` doesn't contain its dependencies (mip resolves
  them at install time), but an MLTBX consumed *without* mip has no
  dependency mechanism at all. Default: dependencies are omitted and a
  warning lists them, for the author to document. With `--vendor`, mip
  flattens the resolved dependency closure into the toolbox so it is
  self-contained (at the cost of duplication, and with the caveat that
  vendoring mixes the dependencies' licenses into the artifact).

Implementation can go through MATLAB's own packaging API
(`matlab.addons.toolbox.ToolboxOptions` + `packageToolbox`) or write the OPC
container directly; the former is less code but constrains the MATLAB
version *of the author running `bundle`* (recipients are unaffected).

### What this MEP is not

- **`.mhl` wrapping an intact `.mltbx`.** Rejected. The channel index needs
  resolved metadata before install, so manifest translation must happen at
  build time regardless — at which point packing the extracted files is
  strictly simpler than nesting containers. The only capability an intact
  MLTBX at install time would add is registering with the Add-On manager,
  which is rejected next.
- **Installing via the Add-On manager.** mip will not call
  `matlab.addons.install` on ingested MLTBX files. The Add-On manager
  permanently edits the path and tracks its own state; mip's model is owning
  `load`/`unload` itself. Routing installs through it would create two
  owners of the same package. (An opt-in "register with Add-On Explorer"
  could be proposed separately if there is demand.)
- **Making MLTBX the primary distribution of mip itself.** Per the community
  feedback: convenient to *offer*, wrong to *require*. A `mip.mltbx` built by
  §3 can sit alongside the existing one-line installer.

## Backward compatibility

Purely additive: a new accepted input format for `mip install` and channel
builds, and a new flag on `mip bundle`. No existing command, format, or
layout changes. `.mltbx` support in `mip install` degrades cleanly on older
mip versions (unrecognized input, clear error).

## Alternatives

- **A dedicated `mltbx/<name>` FQN category** (alongside `local`, `fex`,
  `web`). Rejected: those categories record *where a package came from*;
  MLTBX is a *format*, obtainable from disk or the web. A format-based
  category would double the namespace for no resolution benefit.
- **Delegating ingestion to `matlab.addons.install`.** Gets Add-On Explorer
  visibility for free, but splits ownership of path and state — see above.
- **Export-only or ingest-only.** Each half is independently useful, but the
  boundary story (and the manifest-translation code) is shared; doing both
  at once is little extra work.

## Open questions

- **Vendoring default.** Should `--vendor` be the default for `--toolbox`
  export? A toolbox that errors until its (mip-only) dependencies are
  hand-installed defeats the purpose of the format; but silent vendoring has
  size and license implications.
- **Multi-architecture MEX.** `.mhl` files are per-architecture; a
  distributable MLTBX ideally carries binaries for all supported platforms.
  Should `mip bundle --toolbox` be able to merge several per-arch builds into
  one fat toolbox (as CI could), or is current-arch-only acceptable at
  first?
- **File Exchange overlap.** mip's existing `fex/` source fetches File
  Exchange packages, and some FEX entries are served *as* MLTBX. Should
  `fex/` installs transparently use §1's translation when the download turns
  out to be an MLTBX?
- **GUID scheme.** UUID5 of the FQN is deterministic across machines, but
  should a hand-authored GUID in `mip.yaml` be able to override it (for
  toolboxes that predate mip and already have an identity in the Add-On
  ecosystem)?
- **Manifest fidelity.** Icons, screenshots, examples, and documentation
  entries in the MLTBX manifest have no `mip.yaml` counterpart. Ignore,
  or grow optional `mip.yaml` fields for export?
